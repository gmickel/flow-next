```

- Seed entries via flowctl memory add:
  - 2 active entries (bug/runtime-errors, knowledge/conventions)

=== Cases ===
Case 1: mark-stale roundtrip
  flowctl memory mark-stale <id> --reason "module renamed"
  → entry has status: stale, last_audited: <today>, audit_notes: "module renamed"
  → body unchanged (verify by diff)

Case 2: mark-stale with --audited-by
  flowctl memory mark-stale <id> --reason "X" --audited-by "/flow-next:audit"
  → audit_notes contains "X" + "(audited-by: /flow-next:audit)"

Case 3: mark-stale --json output shape
  flowctl memory mark-stale <id> --reason "X" --json
  → assert success: true, id present, last_audited present

Case 4: mark-stale missing --reason errors
  flowctl memory mark-stale <id>  (no --reason)
  → rc != 0, stderr mentions "required" or similar

Case 5: mark-stale idempotent
  mark-stale --reason "first"; mark-stale --reason "second"
  → second succeeds, audit_notes now "second", last_audited stamped twice

Case 6: mark-fresh roundtrip
  After Case 1's mark-stale, run mark-fresh <id>
  → status cleared, audit_notes empty/cleared, last_audited stamped

Case 7: mark-fresh on active entry no-op-ish
  mark-fresh on never-stale entry
  → rc=0, last_audited stamped, status still active

Case 8: search --status active default excludes stale
  Mark one entry stale, run flowctl memory search "<query in stale entry>"
  → stale entry NOT in results

Case 9: search --status stale includes only stale
  Same setup, flowctl memory search "<query>" --status stale
  → only stale entries returned

Case 10: search --status all includes both
  flowctl memory search "<query>" --status all
  → both active + stale returned

Case 11: list --status stale (existing fn-30 behavior, no regression)
  flowctl memory list --status stale
  → returns stale entries (verify fn-30 path still works)

Case 12: schema validation accepts new fields
  Manually write entry with last_audited + audit_notes in frontmatter
  flowctl memory read <id> → succeeds, frontmatter round-trips intact

Case 13: Ralph regression — mark-stale + mark-fresh + search work under FLOW_RALPH=1
  FLOW_RALPH=1 flowctl memory mark-stale <id> --reason "X" → rc=0
  FLOW_RALPH=1 flowctl memory search "<q>" --status all → rc=0
  (Pure plumbing, no Ralph gate. Skill is gated, plumbing is not.)
```

### `CHANGELOG.md` entry (insert above 0.36.0)

```markdown
## [flow-next 0.37.0] - 2026-04-XX

### Added
- `/flow-next:audit` — agent-native skill that walks `.flow/memory/`, reviews each entry against the current codebase, and decides per entry whether to Keep / Update / Consolidate / Replace / Delete. Interactive mode (default) asks via blocking-question tool; autofix mode (`mode:autofix` argument) applies unambiguous actions and marks ambiguous entries as stale. Inspired by upstream `ce-compound-refresh` skill — adapted for the categorized memory schema shipped in 0.33.0.
- `flowctl memory mark-stale <id> --reason "..."` — sets `status: stale`, stamps `last_audited`, records `audit_notes`. Atomic via existing `write_memory_entry`. Body never modified.
- `flowctl memory mark-fresh <id>` — clears stale flag, stamps `last_audited`. Idempotent on already-active entries.
- `flowctl memory search` accepts `--status active|stale|all` (default `active`), mirroring `memory list`. Stale entries flagged by audit are excluded from default search results.

### Changed
- `MEMORY_OPTIONAL_FIELDS` extended with `last_audited`, `audit_notes`. `MEMORY_FIELD_ORDER` and `_MEMORY_QUOTED_STRING_FIELDS` updated. Validator unchanged (allowed-fields union picks up additions).

### Notes
- `/flow-next:audit` skips legacy flat-file memory entries (`pitfalls.md`, `conventions.md`, `decisions.md`) with a warning recommending `flowctl memory migrate` first.
- Subagent dispatch in the skill is documented for Claude Code (`Agent` + `Explore`), Codex (`spawn_agent` + `explorer`), and Droid; orchestrator falls back to main-thread investigation when subagent primitives are unavailable.
- The audit is fully agent-native — no codex/copilot subprocess dispatch, no Python audit engine. The host agent does the investigation directly using its existing Read/Grep/Glob tools.
```
