# fn-68 pilot backlog mode — full-auto backlog triage + dep-scheduling + async gap-surfacing

## Goal & Context
<!-- scope: business -->

The autonomy track already ships **pilot** (ready spec → plan → reviews → work → draft PR) and **land** (draft PR → CI-fix → gated merge → release), both driven by host `/loop` / `/goal`. But pilot's consent boundary is explicit — *"human judgment lives before pilot"*: it only selects from the **already-ready** queue and assumes specs are triaged, dep-clear, and unambiguous. Everything **in front of the ready gate** — enumerating the whole open backlog, triaging raw items, checking deps, deciding what's next, and unblocking the things that need a human — is still manual prompting today. flow-next is already usable that way (semi-auto by prompting); the gap is a **standing organ that manages the entire open backlog unattended and only pulls a human in when it genuinely cannot proceed**.

Trigger: the Warp *"factory engineers, not product engineers"* memo (2026-06). Its transferable frame: the job is building the factory, measured by **% of changes shipped automatically / (inference cost + human time)**, with agents doing triage → spec → implement → review → verify → monitor and humans stepping in only where needed. flow-next has the conveyor (pilot/land/Ralph) and a two-way tracker channel (tracker-sync); it lacks the **floor scheduler**, the **async human-in-the-loop valve** (so "stuck" becomes a question, not a stall), and the **efficiency readout**.

This spec adds an **optional backlog mode** to pilot. Per tick: enumerate everything open (flow + tracker), select the top dep-ordered actionable item, **triage** it, and — if it is a workable, written spec — advance it one stage along pilot's existing pipeline `plan → plan-review → work → make-pr (draft)` — **full-auto by default**, surfacing a precise question to the human via the spec + tracker when it cannot safely proceed, and **never merging** (land stays human-gated).

**Backlog mode does not author specs.** Spec authoring (`capture`, conversation→spec; `interview`, interactive Q&A) is **human-gated and upstream** — an agent inventing scope from a one-line ticket is precisely the slop the question-valve exists to prevent. So a ticket without a workable spec is **surfaced as a gap** ("run `/flow-next:capture` or `/flow-next:interview`"), never auto-written. The autonomous span runs only from a *workable spec* onward — exactly pilot's current pipeline. The lifecycle, with consent gates marked:

```
idea → [capture / interview] → plan → plan-review → work → make-pr(draft) → [human review] → resolve-pr → [land / merge]
        ^ human, upstream        └──────────── backlog mode runs this span ────────────┘                ^ human, downstream
```

Triage sits at the front and only **routes**: workable → run the span; not-workable → surface "needs a spec," park; blocked → sequence behind the blocker. It never authors and never merges. (`plan` — spec→tasks — is autonomous and already a pilot stage; spec *authoring* is not.)

Decided **not** to build a separate skill (an earlier "foreman above pilot" proposal was considered and rejected): Gordon's full-auto reframe removed the pre-pilot consent boundary that would have justified splitting. pilot already selects across the backlog and walks a pipeline, so backlog mode is a **leftward extension of the same single-tick conductor**, not a new altitude — one `/loop` target, one verdict grammar, one mental model.

## Architecture & Data Models
<!-- scope: technical -->

- **Backlog mode is a behavior of the existing `flow-next-pilot` skill**, gated by config `pilot.autonomy` (default `ready` = current behavior; `backlog` enables the wide mode), overridable per run by `--backlog` / `--auto`. With the flag off, pilot behavior is unchanged.
- **Single-tick contract preserved.** One invocation = select one item + advance one stage + emit one terminal verdict. `/loop` (Claude Code) and `/goal` (Codex) still own repetition. No within-tick loop over multiple items.
- **Selection widens** from "one ready spec" to "one actionable open item across flow specs + tracker issues, **dep-ordered**" — using `depends_on_epics` + tracker `depRelations` + the `flow-next-deps` graph.
- **Readiness is an explicit signal, not an agent judgment.** Eligibility to enter the autonomous span = the **fn-58 flow ready gate is set** OR the **tracker status is at/beyond the configured `tracker.readyState`** (e.g. moved out of Backlog → Todo). The human managing the board — flagging ready, or dragging the ticket out of Backlog — **is the consent act**. Items lacking the signal are not worked. The agent's spec-completeness read is only a **secondary safety net**: a *promoted-but-thin* item (ready signal present, spec still a stub) is kicked back via `ask` ("you marked this ready but it has no AC") rather than built into a slop PR — it never overrides an explicit ready signal to *force* work, only to *withhold* it pending a gap. Raw Backlog items are the human's to promote; backlog mode does not nag every un-promoted idea.
- **Pipeline extends leftward** with `triage` and `ask` stages **in front of** pilot's existing `plan → plan-review → work → make-pr` — it does **not** add a spec-authoring stage:
  - **triage** — classify the selected item by *explicit state first*: workable (ready signal present + spec complete) / promoted-but-thin (ready signal, stub spec) / needs-spec (no ready signal, vague) / blocked-by-dep / needs-human. Routing only.
  - **ask** — when not-workable or ambiguous, write **Open Questions** into the spec, project them as a tracker comment via tracker-sync, mark park-pending-answer. For a *missing or too-thin* spec the surfaced gap is an explicit "run `/flow-next:capture` or `/flow-next:interview`" — backlog mode never authors the spec itself. (It MAY fill an obvious blank in an *existing* spec; it never authors from nothing.)
  - workable, written, dep-clear specs flow into pilot's existing `plan → plan-review → work → make-pr` unchanged.
- **Backlog-mode workflow lives in its own reference file** under the pilot skill (e.g. `references/backlog-mode.md`) so `SKILL.md`'s core single-tick contract stays thin — separation of concerns at the file level, not the command level.
- **Net-new flowctl plumbing (thin):** a backlog-wide ready scan (`flowctl ready --all`; today `ready` is `--spec`-only) returning open items with triage/dep/ready state; and a per-tick **decision log** (receipt-shaped) recording each action + token cost, which powers metrics and the self-improvement substrate.
- **Multi-tracker is inherited, not built.** Backlog mode talks only to tracker-sync's normalized adapter interface (`fetchIssue`/`writeIssue`/`listComments`/`postComment`/`readStatus`/`setStatus`) — which `adapter-interface.md` already defines as transport-blind ("a new tracker is a new adapter, not a reconcile change"). So coverage of **GitHub / GitLab / Jira / Linear** is a property of the adapter set, auto-detected by the discovery ceremony's probe ladder (extended with `glab` auth + an Atlassian-MCP/Jira-token rung). Zero special setup: prefer the company's existing `gh`/`glab` session, registered Linear/Atlassian MCP, or CI env token; spec-first floor when none is present.
- **tracker-sync is the load-bearing substrate — mostly already built, one parity fix.** Backlog mode reuses tracker-sync's existing machinery wholesale: `create-if-unlinked` (auto-link on first touch), per-event automatic sync, never-block/`sync defer`, the readiness projection (`tracker.readyState` → `ready`), comment append + dedup (the question-valve base), and the no-op floor. The **one gap that must close for the autonomous path** is autonomy-signal parity (R14): tracker-sync's Phase 0 recognizes only `FLOW_RALPH`/`REVIEW_RECEIPT_PATH`, not `FLOW_AUTONOMOUS`/`mode:autonomous` — so a backlog-mode lifecycle sync could reach `AskUserQuestion` and hang. The async-question valve (R15) layers on the existing comment channel with a stable anchor; readiness (R16) reuses status-sync. Net tracker-sync work for fn-68 is small: the Phase-0 parity fix + the question-anchor/answer-detect contract.
- **Merge stays out.** land remains a separate human-gated step; backlog mode never invokes merge/land.

## API Contracts
<!-- scope: technical -->

- **Config:** `pilot.autonomy ∈ {ready (default), backlog}`. Per-run override `--backlog` / `--auto`. Optional `pilot.autonomy.gate: [<class>…]` forces surfacing before action for named classes (e.g. `risky`, `prod-config`).
- **Invocation:** unchanged — `/flow-next:pilot` via `/loop` or `/goal`; flags pass through.
- **Verdict grammar (extends `PILOT_VERDICT`):** `TRIAGED <id> <class>` / `ADVANCED <id> <stage>` (ran one pipeline stage on a ready item — reuses pilot's existing verb) / `ASKED <id> (<n> questions)` / `BLOCKED <id> by <dep>` (ready but dep-unsatisfied, sequenced) / `IDLE` (nothing actionable) / `NEEDS_HUMAN <id> <reason>`. **No `PROMOTED` verb** — the agent never sets the ready flag; promotion is the human's board act. One terminal line for the driver, as today.
- **Question channel — transport-blind, multi-tracker, zero-setup:** spec-first (an `## Open Questions` section, **always written** — the universal floor) projected to whatever tracker tracker-sync has an adapter for. Backlog mode never names a tracker; it calls the normalized adapter (`postComment`/`readStatus`/`setStatus`) and inherits coverage. Target set = **GitHub, GitLab, Jira, Linear** (v1 on the already-shipped Linear + GitHub; GitLab + Jira via follow-on adapter specs), each reached via tracker-sync's detect-best-available ladder, preferring tooling the company already has (no flow-next-specific provisioning): `gh` / `glab` CLI sessions, a registered Linear MCP, or a REST/CI env token. **Jira supports both** — agentic Atlassian MCP (like Linear) and standalone REST-via-PAT (so portcos hitting MCP limits are covered without it). When no transport is detected the run is a **no-op + spec-only questions + a one-line "enable X to mirror" note** — never a crash, never a blocker. Human answers in spec or tracker comment → tracker-sync reconciles → next tick re-triages.
- **`flowctl ready --all --json`:** returns open items `{id, source: flow|tracker, triageClass, blockedBy, ready, readySignal}` where `ready` is the **explicit** signal — `flowReadyGate` (fn-58) set OR `trackerStatus` at/beyond `tracker.readyState` (existing config; e.g. `Todo`) — not an inferred completeness score. `readySignal` records which side asserted it.
- **Decision log:** append + summarize subcommand yielding `{tick, id, action, costTokens}` rows.

## Edge Cases & Constraints
<!-- scope: technical -->

- **Never merges** — hard boundary; land stays human-gated.
- **Never answers its own questions**; never promotes an item with an unanswered open question.
- **Park is never silent** — always a logged dep-wait or a surfaced question.
- **Dep deadlock / cycle:** detect, surface as a question, do not spin.
- **Idempotent surfacing:** re-triaging an already-asked/parked item must not duplicate questions/comments (dedup via tracker-sync comment dedup + stable Open-Questions anchors).
- **No autonomous spec authoring:** backlog mode never runs `capture`/`interview` and never writes a spec from a bare ticket; a missing/too-thin spec is surfaced as a "needs capture/interview" gap and parked. Augmenting an obvious blank in an *existing* spec is the only writing it may do.
- **Tracker unavailable / not configured** (no adapter detected, offline, or company runs no tracker): degrade to **spec-only** questions + a one-line "enable `gh`/`glab`/Linear-MCP/Jira-MCP to mirror" note; never crash (mirrors tracker-sync noop). This is the zero-setup floor — the loop is fully functional with no tracker at all.
- **Default off:** existing pilot/land/Ralph users unaffected until they opt in.
- **Single-tick discipline preserved:** the host primitive loops, not the skill.
- **Cross-platform:** canonical Claude names; `sync-codex.sh` regenerates the Codex mirror; `/goal` (Codex) driver parity.

## Acceptance Criteria
<!-- scope: both -->

- **R1:** pilot gains a backlog mode gated by config `pilot.autonomy` (default = current ready-only behavior), overridable per run by `--backlog` / `--auto`; with the flag off, pilot behavior is unchanged.
- **R2:** In backlog mode, one tick enumerates the full open set (flow specs + tracker issues), selects the top dep-ordered actionable item, triages it, and — for a workable written spec — advances it exactly one stage of pilot's existing pipeline (`plan → plan-review → work → make-pr`), emitting one terminal verdict line. The autonomous span runs only from a workable spec onward; `make-pr` (draft) is its terminus.
- **R3:** Triage gates eligibility on the **explicit readiness signal** — the fn-58 flow ready gate set OR tracker status at/beyond `tracker.readyState` — never on an inferred completeness score; items lacking the signal are left for the human to promote. It classifies and **routes**: workable → select-and-advance (pilot's existing path); ready-but-thin → `ask` (kick back, don't build); needs-spec / needs-human → ask; dep-unsatisfied → sequence (`BLOCKED`). It never authors a spec and never sets the ready flag itself.
- **R4:** When it cannot safely proceed, it writes Open Questions into the spec (always) and projects them as a tracker comment via tracker-sync's transport-blind adapter — covering **GitHub, GitLab, Jira, Linear** through the zero-setup detect-best-available ladder (no flow-next-specific provisioning required) — then parks-pending-answer and moves on. No transport detected ⇒ spec-only + note, never a block. Never asks interactively.
- **R5:** Full-auto by default — workable, dep-clear, unambiguous items are selected-and-advanced with no pre-gate (the agent never sets the ready flag itself); a force-gate config can require surfacing before action for named classes.
- **R6:** Never merges; land remains a separate human-gated step; backlog mode never invokes merge/land.
- **R7:** Idempotent surfacing — re-triaging a parked item does not duplicate questions/comments; an answered question causes the next tick to re-triage and proceed.
- **R8:** A backlog-wide ready scan exists (`flowctl ready --all` or equivalent) returning open items with triage/dep/ready state; pilot consumes it for selection.
- **R9:** A per-tick decision log records each action (promote/spec/ask/dep/advance/escalate) + token cost, yielding factory metrics: % moved with no question / one async answer / parked, and cost per change.
- **R10:** Verdict grammar extended (`TRIAGED`/`ADVANCED`/`ASKED`/`BLOCKED`/`IDLE`/`NEEDS_HUMAN` — no `PROMOTED`; the agent never promotes) and documented for `/loop` + `/goal` drivers.
- **R11:** Backlog-mode workflow lives in its own reference file; `SKILL.md`'s core single-tick contract stays thin; docs updated (pilot `SKILL.md`, `docs/ralph.md`, flow-next.dev page + BOTH navbars + changelog); plugin version bumped.
- **R12:** Cross-platform parity — canonical Claude names; `sync-codex.sh` regenerates cleanly; `/goal` driver parity verified.
- **R13:** Multi-tracker, zero-setup, **Linear + GitHub first**. Backlog mode's surfacing is transport-blind across **GitHub, GitLab, Jira, Linear** — it calls only tracker-sync's normalized adapter, never tracker-specific code. **v1 ships on Linear + GitHub** (already-shipped adapters); **GitLab + Jira are follow-on, delivered by their own tracker-sync adapter specs** that fn-68 layers on once present — fn-68 is not blocked on them. Each tracker resolves via the discovery-ceremony probe ladder, preferring auth the company already has — `gh`/`glab` CLI session, registered Linear MCP, or a **CI/REST env token** — with **no flow-next-specific provisioning, OAuth app, webhook, or special config required on the company's side**. **Jira supports both transports like Linear** — Atlassian MCP driven agentically + REST-v3-via-PAT — with the **REST/PAT path standalone so portcos who hit Atlassian-MCP limitations are covered without requiring the MCP** (fn-70). The spec-first floor guarantees the loop works with zero trackers configured; the mirror auto-lights per detected transport.
- **R14 — tracker-sync is autonomous-safe on the pilot/backlog path:** every per-tick tracker interaction runs fully unattended — **never `AskUserQuestion`**. tracker-sync MUST recognize the pilot/backlog autonomy signal (`FLOW_AUTONOMOUS=1` / `mode:autonomous`) exactly as it already recognizes `FLOW_RALPH`/`REVIEW_RECEIPT_PATH`: the discovery ceremony never prompts (no-op + receipt if the bridge is unconfigured), and genuine conflicts / id collisions / readyState-label failures **`sync defer` (queue for the human) instead of asking**. *Concrete fix in scope:* tracker-sync Phase 0 currently gates only on `FLOW_RALPH`/`REVIEW_RECEIPT_PATH` ([steps.md:18](../../plugins/flow-next/skills/flow-next-tracker-sync/steps.md)) — extend it to the full autonomy marker family, matching `work` / `make-pr` / `resolve-pr` / `capture`. (It is the one lifecycle-participating skill whose autonomy gate omits `FLOW_AUTONOMOUS`.)
- **R15 — the async-question valve is an idempotent two-sided round-trip:** the `ask` stage posts Open Questions as a tracker comment with a **stable question anchor** so re-triaging a parked item never duplicates it (rides comments-sync dedup), and the human's answer is detectable from **either** side — an edit to the spec's `## Open Questions` section **or** a reply on the tracker comment (reconcile pulls the tracker-side answer back). An answered question makes the next tick re-triage and proceed; an unanswered one stays parked, no re-post.
- **R16 — readiness-from-status is fresh each tick:** the explicit readiness signal reads tracker status through tracker-sync's existing readiness projection (`tracker.readyState` → local `ready` flag), refreshed on the tick's pull — so a human moving a ticket out of Backlog is reflected on the **next tick** with no manual sync. Reuses status-sync's readiness projection; no new mechanism.
- **R17 — intuitive + automatic, no added setup:** backlog mode introduces **no new tracker setup step**. In steady state the bridge is zero-ceremony — auto-link on first lifecycle touch (`create-if-unlinked`), per-event automatic sync, never-block/defer, no-op floor. If the bridge is inactive, backlog mode degrades to the **spec-first floor** (questions in the spec) with a one-line "enable tracker-sync to mirror" note — never a blocker, never a forced ceremony mid-loop.

## Boundaries
<!-- scope: business -->

- **Not a new skill / command** — a config-gated mode on pilot. The separate-skill ("foreman") proposal is explicitly rejected.
- **Not a loop runner** — `/loop` and `/goal` own repetition; backlog mode is a smarter tick, not a new primitive.
- **Never merges** — land stays human-gated; the merge consent boundary is untouched.
- **Not prospect** — prospect invents *new* work; backlog mode manages the *existing* open backlog.
- **Not interactive interview** — surfacing is async (spec + tracker), never a blocking prompt.
- **Does not author specs** — `capture`/`interview` are human-gated prerequisites that must run before a ticket is workable; triage surfaces "needs a spec" as a gap and parks, it never auto-writes one (may augment an obvious blank in an existing spec only). The autonomous span is *workable spec → draft PR*, not *ticket → draft PR*.
- **Readiness is the human's explicit signal, not the agent's opinion** — the fn-58 ready gate + tracker workflow state (out of Backlog / at `tracker.readyState`) decide eligibility. Promoting a board item *is* consent; the agent never promotes on its own read of the prose, and never works an un-promoted Backlog item. Its completeness check can only *withhold* (kick a thin-but-promoted item back with a question), never *force*.
- **Does not implement tracker adapters** — backlog mode requires GitHub/GitLab/Jira/Linear coverage but consumes it transport-blind through tracker-sync. GitHub + Linear adapters exist; **GitLab and Jira adapters are separate tracker-sync (fn-52 lineage) specs that fn-68 depends on**, because adapters serve every projection (capture/interview/make-pr too), not just backlog mode. Building them inside fn-68 would misplace shared infrastructure.
- **Carries the tracker-sync autonomy-parity fix, not a tracker-sync rebuild** — fn-68 ships the small Phase-0 autonomy-signal fix (R14) + the question-valve contract (R15) because backlog mode is what exposes them; the rest of tracker-sync (auto-link, per-event sync, readiness/dep projection, defer discipline) already works and is reused as-is. Broader tracker-sync UX polish beyond making the autonomous loop correct is out of scope here.
- **Self-improvement *synthesis* is a follow-on** — this spec ships the decision-log substrate + metrics readout; the agent that mines the log into spec-template / plan patches is a separate, later spec.

## Decision Context
<!-- scope: both -->

### Motivation
<!-- scope: business -->

The Warp *"factory engineers, not product engineers"* memo (2026-06): build the factory, measure **% shipped automatically / (inference + human cost)**, agents do triage → spec → implement → review → verify → monitor with humans stepping in only where needed. flow-next already has the conveyor (pilot/land/Ralph) and the two-way tracker channel (tracker-sync); the gaps are (a) a standing backlog scheduler in front of the ready gate, (b) an async human-in-the-loop valve so "stuck" becomes a question not a stall, (c) a factory-efficiency readout. The seductive-but-rejected part of the memo ("1300 ready issues — just let an agent rip") is precisely what flow-next's gated discipline guards against; this spec keeps the merge gate human and makes the *only* erosion of the consent boundary an **async, auditable question**, not an open-loop fire-hose.

### Implementation Tradeoffs
<!-- scope: technical -->

- **Mode-on-pilot over separate skill:** the full-auto reframe removed the consent boundary that justified a split; pilot already selects across the backlog and walks a pipeline, so this is a leftward pipeline extension, not a new altitude. One `/loop` target, one verdict grammar, one mental model. Cost: pilot's `SKILL.md` grows — mitigated by a dedicated reference file. (Earlier "foreman above pilot" considered and rejected.)
- **Full-auto default + async-surface valve over propose-and-confirm gate:** a pre-gate would duplicate what prompting already gives; surfacing-on-block is the memo's semi-automation realized as a conversation, and yields the human-touch-point log for free.
- **Surface "needs a spec" over autonomous authoring:** an earlier draft had triage auto-draft a spec for thin tickets. Rejected — `capture`/`interview` are human-gated by nature (capture needs a conversation; interview is interactive), and autonomous scope-invention is exactly the slop the valve guards against. The narrower span (workable spec → draft PR) is both safer and thinner to build; spec authoring stays upstream and human.
- **Explicit readiness signal over agent-judged completeness:** readiness is the fn-58 ready gate + tracker workflow state (out of Backlog / at `tracker.readyState`), which the bridge already models — *not* the agent reading prose and deciding "good enough." This keeps the human in control via normal board management, avoids the agent gatekeeping subjectively, and reuses existing config. The agent's completeness read survives only as a withhold-and-ask safety net on promoted-but-thin items, never as the promote decision.
- **Multi-tracker via the existing transport-blind interface, not bespoke per-tracker code in pilot:** GitHub/GitLab/Jira/Linear coverage rides tracker-sync's normalized adapter — the cheapest, most maintainable path and the one the adapter interface was designed for. Zero-setup is achieved by *detecting* the company's existing auth (CLI session / MCP / REST token) rather than *requiring* provisioning, with a spec-first floor so adoption needs no tracker at all. GitLab + Jira adapters are split into their own tracker-sync specs (shared infra; serve all projections) and made follow-ons, not folded into fn-68.
- **Linear + GitHub first:** v1 rides the two adapters that already ship — fastest path to a working backlog loop for the majority of users, and it de-risks fn-68 from the GitLab/Jira adapter timeline. GitLab + Jira extend coverage afterward without touching pilot.
- **Jira via the same agentic ladder as Linear, plus a standalone REST/PAT path:** portco feedback is that the Atlassian MCP is too limited for them, so REST-via-PAT (PAT bearer for Data Center/Server; email + API token basic auth for Cloud) must work entirely on its own. But where an Atlassian MCP is present, it's driven agentically just like Linear's. Both transports are first-class (fn-70) — not API-only; the only asymmetry is that REST/PAT is the always-works floor.
- **Config flag (default) overridable per-run (`--backlog`/`--auto`):** config so `/loop /flow-next:pilot` picks it up without arg-threading; flag for ad-hoc override.
- **Default off:** preserves existing pilot/land/Ralph users; opt into the bigger autonomy.
- **Reuse over rebuild:** pilot (the whole `plan → make-pr` executor), land (merge), tracker-sync (channel), flow-next-deps (graph), receipts (log shape). **The readiness gate already exists** — pilot's Phase 1 SELECT (workflow.md) two-pass-filters on `status==open` + the **fn-58 ready flag** + `depends_on_epics` satisfaction, and emits `NO_WORK` when none qualify. Backlog mode reuses that wholesale and adds only: (a) a **second ready source** — tracker status at/beyond `tracker.readyState` (pilot reads the flow flag only); (b) **acting on SELECT's skip pile** — triage/surface/sequence the not-ready / specless items pilot today silently drops to `NO_WORK`; (c) **enumerating tracker issues with no flow spec** (invisible to `flowctl specs`). Net-new code is just the wide ready scan + the decision log + the `triage`/`ask` handlers — no new gate, no spec-authoring engine.
