# Gate diet follow-up: ancestor-walk receipt honor + suite-output capture fix

## Goal & Context

The fn-103 run was trace-measured against the fn-89 baseline (post-mortem 2026-07-19, memory `delegation-brief-eval-2026-07-19` addendum). The 2.17.0 tracker dispatch fully delivered (~190k tokens off-host, 0s serial cost, 3 context items per dispatch). The 2.18.0 gate diet delivered NOTHING: 7 full suite runs for 2 tasks, zero receipts honored, zero docs-only tiers - gate share unchanged at ~50.6% of wall vs the pre-registered target of <=25%. Three verified causes, two of them fixable mechanically:

1. **Receipts are structurally orphaned (the big one).** `gate check` honors only an EXACT HEAD match, but the work loop's own done-flow always commits `.flow` state 27-45s after writing the receipt - so every receipt dies before any consumer can check it. Live proof: receipt `e76025d5` (written 14:22:42) missed by worker .2's baseline check at 14:27:49 (HEAD had moved to `1c157256`; the diff between them was 3 files, ALL `.flow/memory|specs|tasks/**` - exactly the paths the check's own ignore-set deems non-executable). Cost this run: ~245s serial + ~520s compute + a worker turn-split.
2. **Suite-output capture defect (~1,119s = 15% of the whole pipeline).** Both workers piped the suite through `tail -N`, got trailing test noise instead of the `Ran/OK` summary line, and RE-RAN THE ENTIRE SUITE just to observe greenness (runs 2 and 7 of 7 were pure duplicates). No gate policy addresses this - it is a worker.md prose defect.
3. **Force-full floor (accepted, documented, NOT changed here).** Run 6 classified FULL because the diff touched skill prose + its codex mirror - which is correct conservatism (prose-contract tests pin skill wording). Even with fixes 1-2, the floor is ~4 legitimate runs/2 tasks (~35% share); the bigger lever below that is making runs cheaper (parallelization), which is out of scope here and noted as a candidate investigation.

## Acceptance criteria

- **R1: Ancestor-walk receipt honor.** When the exact-HEAD filename lookup misses, `gate check` globs `GATE_RECEIPTS_DIR` for `*-<gate_id>.json` candidates and evaluates them under a DETERMINISTIC policy: sort by the receipt's embedded `timestamp` DESC with the FILENAME as a stable tie-breaker (python sort - glob order is fs-dependent); candidates whose timestamp is missing/unparseable/not-timezone-aware are dropped BEFORE ranking; evaluate AT MOST the 8 newest, short-circuit on first honor. A candidate honors iff: internal consistency (filename sha8 == `head_sha[:8]` - a copied/renamed receipt is skipped); NOT a symlink (per-candidate `is_symlink()` - the existing single-file guard does not cover a glob loop); `head_sha` is an ancestor of HEAD (`git merge-base --is-ancestor`, rc 0; rc 1 = not-ancestor -> skip; any other rc/OSError -> skip, fail-closed - genuine repo breakage was already screened by `_gate_repo_and_head` which exits 2); the TWO-dot `git diff --name-only -z --no-renames <receipt_sha>..HEAD` contains ONLY ignore-set paths (bare diff paths fed directly to the SAME `_gate_ignored_worktree_path` predicate - one shared helper, no second list; any non-ignore-set path -> skip); AND the shared validator conditions hold. STRUCTURE: the existing inline checks TERMINATE via `_gate_exit_check`, so they cannot be re-entered per-candidate - extract a PURE return-based validator (`_gate_receipt_valid(receipt, head_sha, command) -> (ok, reason)`) covering schema / command-fingerprint / `0<=age<=24h` / full-`head_sha`-vs-expected-sha identity (the expected sha is HEAD in exact mode, the candidate's own head_sha in walk mode), plus a SEPARATE return-based helper for the ancestor-mode filesystem/git checks (`_gate_walk_candidate_ok(receipt_path, receipt, repo_root, head_sha) -> (ok, reason)`: filename sha8 consistency, symlink refusal, ancestry, two-dot ignore-set diff); BOTH the exact path and the walk consume the pure validator, the walk additionally runs the candidate helper (exact path behavior byte-equivalent, same reasons, same exit taxonomy). Worktree cleanliness is checked ONCE per invocation (not per candidate), preserving the exit-2 taxonomy for real `git status` failures. Malformed/corrupt candidates are skipped, never abort the walk. No candidate honors -> exit 1 ("no honorable receipt for HEAD"). No new flags; no commit-count bound (the 8-candidate cap bounds the EXPENSIVE git evaluations; scan I/O reads every candidate for ranking, which stays small because `gate receipt` now PRUNES on write: best-effort delete of receipts in the dir older than 24h - they are unhonorable by TTL anyway - so accumulation is bounded at ~a day of receipts). Receipts never travel to CI (`.flow/tmp` is gitignored), so shallow-clone environments simply never enter the walk.
- **R2: Capture fix.** worker.md's suite-run prose captures greenness from the COMMAND EXIT CODE (`; echo "suite_rc=$?"`) with output to a file (`> log 2>&1`), reading any summary from the file - NEVER re-running a suite to observe its result. The rule is stated ONCE and applied at both real suite-invocation sites: Baseline check (~L52) and Verify-before-completing (~L390); one clarifying sentence records that DELEGATED tasks route through the SAME Verify block (no separate delegation test site exists in worker.md). The delegate's OWN in-sandbox suite runs are governed by the fixed path-handoff template - add one clause to its verify slot in codex-delegation.md ("observe greenness via the command exit code, never by re-running"). Sweep both files for any other run-and-observe site. Rule sentence: a green observation is the exit code, not a scraped line; re-running a suite for observation is forbidden.
- **R3: Tests + exact gate commands.** Quick commands (verbatim, these are the receipt-identity strings): focused - `cd plugins/flow-next/tests && python3 -m unittest test_gate_receipt test_gate_classify -q`; full - `cd plugins/flow-next/tests && python3 -m unittest discover -q`; mirror idempotency - run `./scripts/sync-codex.sh`, snapshot (`git diff -- plugins/flow-next/codex | sha256sum`), run it again, snapshot again: the two snapshots MUST be identical (the first run may legitimately generate uncommitted mirror changes; idempotency = the second run adds nothing); dual-copy - `cmp plugins/flow-next/scripts/flowctl.py .flow/bin/flowctl.py`. Test matrix: New `GateReceiptAncestorWalkTestCase(GateReceiptHarness)`: ancestor honor (receipt at sha A, .flow-only commit -> exit 0); non-ignore-set commit in the walk -> exit 1; DIVERGED sibling-branch receipt -> exit 1 (not just older-behind); filename/body sha mismatch -> skipped; corrupt candidate JSON does NOT abort the walk (a later valid candidate still honors); symlinked candidate skipped; candidate-cap respected with deterministic newest-first order (write >8 receipts, only newest 8 evaluated); command-fingerprint mismatch on one candidate -> next candidate tried; existing exact-match tests unchanged (incl. test_check_rejects_moved_head - still exit 1, docs path is non-ignore-set).
- **R4: Measured claim discipline.** CHANGELOG + docs state the honest status: the diet's receipt path was structurally dead in 2.18.0, fixed here; the ~35% floor and the parallelization lever are recorded in the post-mortem decision context, not promised.
- **R5: Cross-platform.** Dual-copy flowctl, mirror x2 idempotent, no version bump (batched).

## Decision Context

- R3 (receipt-after-state-commit ordering) was considered and DROPPED pre-review: the ancestor-walk makes ordering irrelevant (a receipt behind N .flow-only commits is honored anyway), and reordering the done-flow adds prose complexity for no additional coverage. Trimmed 2026-07-19 before the first review cycle (renumbering safe).

- Post-mortem evidence (fn-103 run): 7 suite runs / 2 tasks; 3 receipts written, 0 honored, all orphaned by .flow state commits 27-45s later; duplicate runs 2+7 = ~1,119s pure waste from tail-scraping; tracker dispatch delivered exactly as designed (~190k tokens off-host, 0s serial).
- The ancestor-walk predicate is hash + path membership - mechanical, doctrine-clean (STRATEGY.md design principles: flowctl burden of proof met; no judgment).
- Force-full conservatism for skill prose is KEPT (prose-contract tests pin wording; loosening it trends toward semantic judgment).
- Suite parallelization is the next lever below the ~35% floor - separate investigation, not this spec.
- fn-109 (PR #215, repo-root memoization) lands concurrently: verified fully disjoint (hunks 11k+ lines away; gate section deliberately bypasses memoized helpers with fresh git reads). Only textual CHANGELOG overlap - merge main before release.
- Downstream handoff (named, separate repo): flow-next.dev documents `gate check` / Green receipt vocabulary - apply the ancestor-clause update in ~/work/flow-next.dev in the landing workstream (fn-103 precedent); record the handoff in the done summary so the lander walks it.
- Pruning semantics (chosen, in scope): `gate receipt` prune-on-write - after a successful receipt write, best-effort delete of sibling receipts in the dir whose embedded timestamp is older than 24h (they are TTL-unhonorable); failures to prune are silent (never block the write). Accumulation is thereby bounded at ~a day of receipts; the 8-candidate cap bounds per-check git work on top.
- Delegation A/B from the same run: path-handoff composition 244s net vs 519s composed-brief (2.1x faster), prompt 13x smaller - fn-103 confirmed in production.
