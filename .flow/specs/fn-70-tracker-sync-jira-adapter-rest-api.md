## Goal & Context
<!-- scope: business -->

tracker-sync ships **Linear** and **GitHub** adapters; **Jira is detected but deliberately not offered**. Enterprise teams overwhelmingly run Jira and want flow specs mirrored there. This spec adds a Jira adapter implementing tracker-sync's normalized interface so `/flow-next:pilot` backlog mode (fn-68) and every other projection can target Jira with **no special setup beyond a credential the company already issues** â€” covering Jira **Cloud and self-hosted Data Center/Server**.

**Transport decision (2026-06-28): the Jira REST API + token is the sole supported transport** (Cloud `/rest/api/3` + API-token, DC/Server `/rest/api/2` + PAT) â€” a single-rung + no-op adapter shaped like the GitHub adapter, NOT a Linear-style MCP ladder. The Atlassian MCP was evaluated and is **not wired**: the *official* Atlassian Remote MCP (GA Feb 2026) is read-mostly â€” it **cannot transition issue status, update existing fields, or set issue/epic links**, exactly the writes a two-way sync needs; the *community* MCP (`sooperset/mcp-atlassian`) is a REST-wrapper needing its own Docker server + the same PAT, so it adds setup for no capability flow-next's native REST path lacks. (This supersedes the earlier "support MCP too, like Linear" intent, on the evidence.)

- **Relations â€” native links, `linkPresent` always true (fn-69 invariant).** Jira issue links are **native and not license-gated** (no GitLab-style `<!-- flow:deps -->` body-block fallback). - **Transport: the Jira REST API + token, single rung + no-op** (GitHub-shaped). - **Auth (two schemes â€” detected at ceremony, persisted):** Jira **Cloud** = HTTP basic `email:API_TOKEN` (`JIRA_EMAIL`+`JIRA_API_TOKEN`, an *API token*, not a PAT); Jira **Data Center/Server** = `Authorization: Bearer <PAT>` (`JIRA_PAT`). Label unambiguously so a Cloud user isn't told to make a non-existent "PAT". **Self-hosted TLS:** opt-in `JIRA_SSL_VERIFY=false` for internal-CA/self-signed certs (documented, never silent). Never store creds in flow state â€” read env each run.
- **Backlog enumeration (`listOpenIssues`) â€” endpoint by `apiVersion`:** **Cloud v3** = **`POST /rest/api/3/search/jql`** (the legacy `GET /rest/api/3/search` was **removed**, HTTP 410, CHANGE-2046; verified live 2026-06-28) + cursor pagination (below); **DC/Server v2** = `/rest/api/2/search`, whose endpoint + pagination model (likely **offset** `startAt`/`maxResults`, not cursor) are **verify-at-build** â€” the Cloud 410 evidence does NOT prove DC behavior; DC `listOpenIssues` **IS implemented** via `/rest/api/2/search` (verified at build against a real DC instance), **not deferred** (R1/R9 require all nine methods for DC). Branch on `apiVersion`. Scope JQL to the project (Free-tier rate limits). **JQL safety:** the raw `readyState` status name is **escaped** (quotes/backslashes) before interpolation into `status = "<readyState>"`, and `projectKey` is **validated against the Jira key grammar** before building the query â€” no JQL injection.
- **ADF body translation** â€” Jira REST v3 descriptions/comments use Atlassian Document Format. The adapter translates normalized Markdown â†” ADF at the boundary (Jira-specific; reconcile sees only normalized text). Lossy-but-faithful over a documented subset. **Unknown-node preservation algorithm (required, R3):** never blind-regenerate ADF from markdown (that drops unknown nodes). Instead, on a body write **fetch the current ADF first**, splice the new content **only into the regions that map to the supported markdown subset**, and **carry unknown/unsupported ADF nodes through verbatim** (treat opaque ADF as metadata outside reconcile). (Verified 2026-06-28: description + comment bodies are `{"type":"doc","version":1,"content":[...]}` and round-trip exactly on the supported subset.)
- **Comment `authorAuthority` (fn-68 contract â€” security-sensitive for the answer valve):** `comment.author.accountType` gives a coarse signal â€” `app` â‡’ `bot`, `customer` (JSM portal) â‡’ `outsider` â€” but **`atlassian` does NOT automatically mean `writer`** (an internal Atlassian account isn't necessarily a project writer/approver). For the answer valve, grant **`writer` only with a project-role/group check** (`GET /project/{key}/role` membership) **OR** membership in the configured `tracker.answerAuthors` allowlist; otherwise **`unknown`** (fail closed). **Do NOT introduce a new enum value** â€” the normalized `authorAuthority` enum is exactly `writer|outsider|bot|unknown`, so a non-authorized Atlassian account maps to `unknown`. The comment author object carries `accountId`/`accountType`/`displayName` but **no inline role** (verified 2026-06-28), so the role lookup is a separate call.
- **Status fidelity â€” workflow-aware + fn-66 terminal invariant.** Jira has rich per-project workflows. **Status WRITES** map the normalized flow status â†’ a Jira status via `statusMap`, applied through the **transitions API** (`GET /issue/{issueIdOrKey}/transitions` â†’ `POST /issue/{issueIdOrKey}/transitions {transition:{id}}`, preferring the durable `tracker.id`; verified 2026-06-28), never a direct set. **Terminal detection = `status.statusCategory.key`** (`new`|`indeterminate`|`done`), NOT the status *name* (names are project-renamable). Honor the fn-66 invariant: locally-`done` â†’ **In Review** until merge, terminal **Done** gated on `MERGED` PR evidence. **Not every project has the assumed statuses** â€” the default team-managed workflow is To Do/In Progress/Done with **no "In Review"** (verified 2026-06-28); a missing `statusMap` target or an unreachable transition â‡’ **defer + receipt** (the deterministic default), never assume the In-Review rung exists. **"Nearest" is allowed ONLY via an explicit configured fallback** â€” never inferred (an inferred nearest risks an illegal/over-eager transition). **`readStatus` (READ normalization â€” the inverse of the write-map):** reverse-map the Jira status â†’ a normalized status via `statusMap` (match by `id` first, then `name`); else fall back to `statusCategory.key` (`done`â†’`done`; `new`/`indeterminate`â†’a safe nonterminal **only when unambiguous**); otherwise return **unmapped + defer receipt** (never guess). Sharp edge: a target transition may be unreachable from the current state â†’ defer + receipt, never a forced/illegal jump.
- **Activation gate (flowctl):** extend `TRACKER_TYPES` to include `"jira"` (deterministic enum edit â€” the single line that activates `tracker.type: jira`).
- **PRâ†”issue link (make-pr):** projects to Jira as a **remote link** (or a URL comment) in-adapter â€” no auto-linkify/`gh`; Smart-Commit keys not relied on.

## API Contracts
<!-- scope: technical -->

- **Detection (ceremony, once):** `JIRA_BASE_URL` + (`JIRA_PAT` | `JIRA_EMAIL`+`JIRA_API_TOKEN`) â‡’ REST transport available. Resolution `env > config > ask`; persisted at the ceremony. Surface present AND absent.
- **Auth:** Cloud basic (`email:API_TOKEN`) vs DC/Server Bearer PAT â€” distinct env vars, labelled in `references/jira.md`.
- **Adapter methods:** exact `adapter-interface.md` signatures; ADFâ†”Markdown only inside the adapter.
- **Config:** `tracker.type: jira`; `tracker.perTracker` carries `baseUrl`, `projectKey`, **`authScheme` (`cloud-basic` | `bearer-pat` â€” non-secret; persists the ceremony's deployment decision so runtime never re-infers; credentials still read from env), `apiVersion` (`3` Cloud | `2` DC/Server), optional `sslVerify`,** and `statusMap`. **`statusMap` shape:** a map of **normalized status â†’ an object `{"name": "In Review"}` OR `{"id": "10031"}`** (`id` is preferred over `name` when both are present â€” names are project-renamable), for the FULL normalized set the adapter emits (`planned`/`in-progress`/`in-review`/`done`/`verified`/â€¦), not just ready/done; e.g. `statusMap: {"in-review": {"name": "In Review"}, "done": {"id": "10031"}}`. **`tracker.readyState` is SEPARATE â€” a raw Jira status name** (e.g. `readyState: "Ready for Dev"`) used directly in the promoted-lane JQL filter, **NOT** resolved through `statusMap` (consistent with Linear/GitHub treating `readyState` as the raw tracker-side value). - **Status writes** go through the **transitions API** (resolve transition id for target status, POST transition).
- **Identity:** durable `tracker.id` = the **immutable Jira issue `id`**; `--identifier PROJ-123` is the display key. `sync set-tracker-id <spec> <id> --identifier PROJ-123 --url <browse-url>` â€” the link flow accepts a `key`, resolves it to the `id` before persisting. ## Edge Cases & Constraints
<!-- scope: technical -->

- **No credential reachable** â†’ no-op rung + actionable receipt ("set `JIRA_BASE_URL` + a token"); spec-first floor still works.
- **ADF round-trips are lossy** â€” document the supported Markdown subset; preserve unknown ADF nodes on write-back (no silent loss).
- **Transition gating** â€” a target status may be unreachable from the current status per the workflow; resolve the legal transition, else defer + receipt (never force).
- **Cloud vs Data Center auth differ** â€” detect by token type at the ceremony; support both.
- **Self-hosted TLS** â€” opt-in `JIRA_SSL_VERIFY=false` for internal-CA certs; never silent / never default-off.
- **Rate limits / 401 / 403** â€” bounded retry; auth failure â†’ no-op + actionable receipt; never block. (Free/Standard Cloud tiers have stricter limits â€” scope JQL to the project.)
- **Cross-platform:** canonical Claude names; `sync-codex.sh` regenerates cleanly.

## Decision Context
<!-- scope: both -->

### Motivation
Enterprise teams run Jira and want flow specs mirrored there. tracker-sync's transport-blind interface means Jira is "just another adapter"; the Jira-specific weight is ADF translation + the transitions-API status model.

### Implementation Tradeoffs
- **Jira REST API + token single-rung, not a Linear-style MCP ladder (decided 2026-06-28, MCP evaluated):** the *official* Atlassian Remote MCP (GA Feb 2026) is read-mostly â€” Atlassian's own community threads confirm it **cannot transition status, update existing fields, or set issue/epic links** (only basic-issue create, read, comment, JQL). A two-way sync needs exactly those writes, so the MCP can't be the transport. The *community* MCP (`sooperset/mcp-atlassian`) wraps the full REST API but is a third-party Docker server needing the same PAT â€” redundant with flow-next's native REST path and itself "special setup." So Jira is GitHub-shaped: the Jira REST API + token, single rung. Reverses the earlier "support MCP too, like Linear" intent on the evidence.
- **Transport resolved once at the ceremony, persisted (like review-backend):** no per-run "is the MCP/token good" probing â€” `env > config`, the no-op rung is the only runtime fallback. Avoids "checking for support all the time."
- **ADF translation in-adapter:** keeps reconcile transport-blind; cost is a documented lossy subset + unknown-node preservation.
- **Transitions API over direct status set:** Jira forbids arbitrary status writes; some target states are unreachable â†’ defer+receipt, never force.
- **Live-API smoke-tested (2026-06-28, dociq.atlassian.net, throwaway project deleted after):** every adapter op validated against real Jira REST v3 â€” create/read/update issue, ADF `doc` round-trip, transitions, `Blocks` link, remote link, JQL search, colon-labels. Surfaced three corrections now folded in: the `/search`â†’`/search/jql` removal (410), `statusCategory.key` as the terminal signal (not status name), and that "In Review" isn't universal. Evidence: `~/work/agent-scripts/flow-smoke/out/FINDINGS-jira.md`.
- **Plain API token over Connect/Forge/MCP:** the zero-setup promise â€” companies already issue API tokens.
