# Opinionated prime mode: project-type & structure-aware readiness (monorepo vs multi-repo, size, mapping)

> Full spec, ready for `/flow-next:plan`. Captured 2026-07-10 from external-team field feedback; research passes 2026-07-11 (9 research agents across two waves, 2025-2026 sources; wave 2: enforcement layers/hooks, agent observability + drivability, systematic rubric gap-diff vs 13 published rubrics, harness-config readiness). Interview intentionally skipped - the open design decisions are made in Decision Context and flagged for review there. Prime is the FIRST skill every user runs - the criteria set below is deliberately ambitious. Today `/flow-next:prime` largely checks file/config **existence**, so repos reach "Level 5" without the substance behind it, and there is no way to hand-verify hundreds of repos across 10-11 tech stacks. This spec makes prime **opinionated + structure-aware**: classify what kind of project this is (greenfield | single repo | monorepo | multi-repo constellation), probe size/legibility and operability with bounded deterministic evidence, judge substance not existence, and lead the report with a verdict + ranked next-actions instead of a level.

## Goal & Context
<!-- scope: business -->

At scale, existence-checks lie. A repo can pass prime's criteria - a CLAUDE.md exists, a hook file exists, a script exists - yet be un-agentic in practice: the file is an empty template, the repo is too big to navigate, imports can't be resolved, the app cannot be compiled by anyone but the IDE, or it's really one of 99 sibling repos that only make sense together. Leads/reviewers can't hand-verify at portfolio scale, so a top "readiness level" stops meaning anything. Prime should encode the judgment a senior applies on a first look - how big is this, can an agent find its way around, can it BUILD anything, is this a monorepo or a swarm of repos, is it a day-old empty shell - and say so **opinionatedly**, with a concrete ranked recommendation list rather than a pass/level.

The same prime run must give grounded, realistic guidance across radically different repos: a modern TS pnpm/Nx monorepo (everything works, deepen the loops), a 15M LoC Delphi codebase (no LSP, no map tooling, encoding hazards, compile-only feedback), and a product spread over 99 sibling repos (per-repo priming stays valid, but a home-base coordination layer is the unlock). One playbook per shape, one matrix across stacks.

Origin: a portfolio-company rollout (three product lines), 100+ repos across ~10-11 stacks, where "häufig nur Existenz geprüft -> Level 5 erreicht, ohne dass es stimmt." Gordon owns the write-up.

## Architecture & Data Models
<!-- scope: technical -->

### Mode shape (decided)

**No new mode or flag.** Classification, substance checks, the operability verdict, and the ranked-actions headline become prime's DEFAULT behavior. The existing 8-pillar / 48-criteria scan stays as the evidence layer underneath; existing args (`--report-only`, `--fix-all`, repo-root path) keep their semantics. Rationale: the old existence-only behavior is exactly what fn-92 exists to retire; keeping it reachable behind a flag preserves the failure mode. The one new arg is `--classify-only` (print the Phase 0.5 classification block and exit; cheap portfolio triage over many repos).

Skill layout: FOUR new reference files keep SKILL.md/workflow.md lean, mirroring the existing pillars.md/remediation.md pattern (flat skill-root siblings, per prime's own precedent):

- `classification.md` - detection signals, thresholds, confidence tiers (Phase 0.5)
- `playbooks.md` - per-shape playbooks (greenfield, standard, TS-style monorepo, huge/legacy, constellation) + report-shape templates + the ranked-actions catalog
- `stacks.md` - per-stack readiness matrix (detect / verify / LSP / map support / gotchas)
- `harness.md` - the Harness & Permissions check set (HP) + per-harness instantiation rows (which file, which syntax, per function class) - same data-not-logic discipline as stacks.md

### Phase 0.5: Classify (new phase, before scout dispatch)

Deterministic, host-inline (no scout), bounded to `git ls-files` / `find -maxdepth` / config-presence / at most one sampled `rg` probe. Produces a classification block that parameterizes everything downstream (scout dispatch hints, N/A denominators, report shape, playbook selection).

**Axis 1 - lifecycle:** greenfield | hybrid ("young but real") | brownfield.
Greenfield = (`git rev-list --count HEAD` < ~25 AND no tags AND no CI config) OR (tracked files < ~30 with no domain source beyond a recognizable generator scaffold - create-next-app, cargo new, uv init, dotnet new fingerprints). Corroborators: repo age, single contributor, no lockfile. Multi-signal, never one heuristic; borderline cases land in hybrid.

**Axis 2 - topology: TWO INDEPENDENT BITS, not an enum** (eval finding: DocIQ-Sphere is a turbo monorepo AND a dociq-* constellation member - an XOR verdict silently drops constellation guidance for every monorepo member).
Bit 1, monorepo = workspace/build-graph config present (`pnpm-workspace.yaml`, package.json `"workspaces"`, `nx.json`, `turbo.json`, `WORKSPACE`/`MODULE.bazel`, `go.work`, Cargo workspace, `.groupproj` for Delphi, multi-module Maven/Gradle) OR many CROSS-REFERENCED manifests under one root - a bare manifest count is not enough (eval finding: a spike + a scaffold + a docs-site subdir produced a false monorepo); spikes/scaffolds/site subdirs are downweighted.
Bit 2, constellation-member, three confidence tiers:
- (a) LIKELY: sibling dirs with `.git` (`ls ../*/.git`), shared org in remotes, shared naming prefix (`acme-api`, `acme-web`). **Workspace-parent dampener (load-bearing - eval-proven):** a parent with many git dirs (>~20) is a developer WORKSPACE, not a home base - shared-org there is meaningless (3 of 5 real eval repos false-positived without this). In a workspace parent, only a prefix-family cluster (>=2 named siblings) or tier-(c) references raise LIKELY, and it always routes to the R15 ask, never auto-confirms. Prefix matches can include scratch/test repos - LIKELY-tier input only.
- (b) CONFIRMED: parent-level CLAUDE.md/AGENTS.md or manifest (`repos.yaml`, `mani.yaml`, `.meta`, `*.repos`, `workspace.toml`, `default.xml`, `.code-workspace`), parent docker-compose/justfile/mise.toml, `_plans/` dir - and NOT a workspace parent.
- (c) ASK THE USER: in-repo external references - compose `build: ../other-service` or same-org images, CI checkout of a second repository, OpenAPI/proto client generation pointing at another repo's spec, `go.mod` replace / pnpm `link:` / `pip -e ../` outside the repo, env vars referencing ports of services this repo doesn't define, AND **textual cross-repo path references in the agent file/README** (eval: both real constellations found were referenced in prose - "see ~/work/transcribe/docs/RELEASING.md (the app repo)" - while the pattern probes returned empty; prose references also sharpen the R15 question per "never ask what a probe answered").
Confidence is a separate field on every axis, never punctuation in the verdict enum.

**Axis 3 - size/legibility band:** small (<100K LOC) | medium (100-400K) | large (400K-2M) | huge (>2M or >20K files).
LOC via `scc` when available (seconds even at 15M LoC), else `tokei`, else file-count x estimate - NEVER `cloc` (8-20x slower), NEVER exhaustive reads.
**Exclusions BEFORE counting (eval-proven, every one hit a real repo):** tool-managed dirs (`.flow/`, `.claude/`, `.codex/`, `.agents/`, `.cursor/`, `.factory/`, `.windsurf/`, `.opencode/`, `.pi/` - flow-next's own bundled `flowctl.py` alone produced a phantom "Python 42%" stack and +40% LOC on a pure-TS repo; prime polluting its own classification is the most embarrassing possible failure), vendored/generated globs, test/eval fixture corpora, agent-workflow state (`plans/`, `history/`), content-hash duplicates (a dual-copy file was 57% of one repo's counted LOC and flipped its band), dirs a tracked script deletes-and-regenerates, and tracked legacy snapshots (an `original-implementation/` dir the repo's own agent file marks off-limits). The same exclusion list feeds the largest-files pathology inventory - prime must never report its own tooling as the repo's top pathology. Legibility sub-signals: top-level dir count, entrypoint-glob hits (`main.*`, `index.*`, `cmd/*/main.go`, `*.dpr`, framework markers), packages-vs-instruction-files ratio, tracked generated/vendored code (`vendor/`, `dist/`, `*_pb2.py`, `.dcu`), root instruction file >~300 lines with zero nested files, one bounded grep-ambiguity probe (2-3 core domain identifiers; hundreds of hits predicts wrong-file/wrong-symbol thrash). Empirical anchors: agent pass rates collapse >100K LOC (RepoMod-Bench); code-intelligence tooling is net NEGATIVE <400K LOC and strongly positive 400K-2M (Sourcegraph CodeScaleBench, 1,281 runs) - recommendations must tier accordingly; recommending heavy tooling on a small repo is itself a prime failure.

**Axis 4 - stack(s):** MANIFEST-GATED detection per `stacks.md` (package.json, pyproject.toml, pom.xml/build.gradle, *.sln/*.csproj + TargetFramework era, composer.json, go.mod, Gemfile, CMakeLists, *.dproj/*.dpr, *.vbp, *.pks/*.pkb, *.cbl), with a post-exclusion LOC-weighted extension histogram as CORROBORATION (a stack with no manifest and no meaningful LOC share never appears; the histogram vocabulary derives from detected manifests + extensions, never a fixed list - eval: a fixed six-stack list made Ruby invisible while phantom Python ranked second). Subproject manifests (site subdir, spike) are noted as subprojects, not top-level stacks. Polyglot count feeds the legibility verdict.

**Axis 5 - delivery shape(s), MULTI-VALUED (eval-driven addition - the single root cause of most misfires):** web service/app | CLI | library/SDK | plugin/prose-product | desktop | docs site | data/ML | IaC. One repo can carry several (a real eval repo is CLI + daemon + web UI + desktop shell). Shape gates: (a) the LOC denominator policy - prose counts as source for plugin/prose and docs-site shapes (a plugin repo measured "Python 92%" while its actual product was 146K lines of markdown the histogram ignored); (b) the operability-ladder CEILING (library/plugin/prose ceiling = tier 2, reported "2/2 at ceiling", never "2 of 3", with move-up suppressed and the cheapest move going SIDEWAYS into observable/drivable instead; desktop shape's honest tier evidence is the repo's own packaged-runtime smoke, not a boot probe); (c) which AO/DR rows apply (CLI-shaped repos grade on the agent-first-CLI row - `--json`, exit codes - as the primary drivability surface); (d) playbook selection. Detection: manifest bin/exports fields, framework markers, serve/health code, .desktop/electron/tauri/electrobun markers, skills/prose ratios - heuristic, confidence-marked, R15-askable.

Classification is heuristic: the block always prints its evidence and confidence.

### Phase 0.6: Targeted clarification (bounded ask protocol)

Some classification facts are LOW-CONFIDENCE (probes disagree) and some are UNINFERABLE IN PRINCIPLE from the working tree alone - org context the filesystem cannot show: do sibling repos exist elsewhere (the agent may see only this checkout)? does a headless-build toolchain/license exist for CI? which subsystems are frozen vs safe agent territory? is this repo's "missing CI" actually run by an external system? Prime asks instead of guessing, under a strict budget:

- **At most one AskUserQuestion call (up to 3 questions)** per run, only for facts that (a) scored low-confidence or uninferable AND (b) change the playbook or a verdict. Never ask what a probe already answered; every question states what WAS inferred, the evidence, and what hangs on the answer (stakes + consequences, per the plain-language question contract).
- **Answers become durable repo facts, not session trivia:** on confirmation, offer to record them where the next run can INFER instead of re-asking - topology/carve-out/toolchain facts into the agent instruction file (a short "Repo context" block: "part of the acme constellation, home base ../acme"; "CI builds via Build.cmd on a licensed runner"; "modules X,Y frozen"), which simultaneously fixes the corresponding DC2 gap. Declined = noted in the report, re-derived next run.
- **Non-interactive (`--report-only`, autonomous/Ralph):** never block; state each assumption inline and emit an "Unresolved questions" list in the report so a human can settle them once.
- `--classify-only` never asks (it is the cheap sweep); it prints confidence + the would-ask list instead.

### Operability ladder (replaces "can the app run" as the baseline)

The single most important substance signal, graded as a LADDER so legacy stacks get honest treatment instead of a Node-centric fail:

| Tier | Meaning | Verify evidence |
|---|---|---|
| 0 static-parse-only | no compiler/runtime reachable (mainframe, licensed compiler absent) | linter/parser passes on a sample |
| 1 compile-only | agent can build the touched project headlessly | build command exits 0 (bounded timeout ~5 min) |
| 2 compile + test subset | console test runner exists and is discoverable | list-tests/dry-run succeeds (existing TS4 machinery) |
| 3 run | app boots with seeded data; smoke one-liner | port bound / health 200 / ready line within ~60s |

The report names the CURRENT tier (from executed evidence, extending the existing Phase 2 verification - never from config existence) and the **cheapest move up one tier** (e.g. "add a console DUnitX target compiled in CI" for a tier-1 Delphi repo; "add `make bootstrap` chaining install -> services -> migrate -> seed" for a tier-2 web app). Never prescribe "start the app" for a stack OR SHAPE whose realistic ceiling is tier 1-2 (shape ceilings per Axis 5: at ceiling, report "N/N at ceiling" and offer sideways moves into observable/drivable). Tier-3 boot probes run only when a cheap ready-signal is detectable (health endpoint, dev-server ready line) and always time-bounded; otherwise the tier is recorded as "not probed", not failed. Two eval-proven qualifiers: **monorepos get PER-MEMBER tiers**, not one repo tier (a real repo was tier 3 in one app and cloud-gated in another); and **tier 3 needs an external-SaaS gate** - a ready line + bound port is NOT tier 3 when the backend of record is a cloud service requiring interactive auth (Convex/Firebase-class repos boot a nonfunctional shell; report "tier 3 gated on <service> credentials" instead of fabricating a pass).

**Hard gates (cap the maturity level regardless of score):** (G1) the detected build command actually runs, or tier >= 1 evidence exists; (G2) tests are discoverable when a test framework is claimed (existing TS4); (G3) commands quoted in CLAUDE.md/AGENTS.md actually resolve/execute. Any gate failing caps agent readiness at Level 2 with the failure named in the headline. This kills "Level 5 with a broken build".

### Substance-not-existence upgrades (specific criteria)

General principle: **execute it (bounded) over parse it; cross-reference it against the code over read it in isolation.** Cross-reference failures (dangling commands, undeclared env vars, CI/local mismatch) are the cheapest high-precision substance detectors, no LLM judgment needed. Concrete criterion changes:

- **SV4 REWRITTEN: "deterministic feedback gate", layer-agnostic (not "pre-commit hooks configured"):** the field evidence is unambiguous that test-running pre-commit hooks are an agent anti-pattern - agents bypass them (`--no-verify` incident anthropics/claude-code#40117: 6 commits, 63 failing tests, despite deny rules AND memory instructions) or stall terminally on them (Copilot coding agent; community workaround is literally "instruct it to retry with --no-verify"). The layered model:
  - L1 edit-time: harness hooks (Claude PostToolUse / Cursor afterFileEdit / Droid PostToolUse) - format + file-scoped lint/typecheck, cannot be `--no-verify`d
  - L2 commit gate: git hooks scoped to format/lint on staged files, <~10s, auto-fix where possible - NEVER test suites
  - L3 task end: the verify command gated by ACCEPTANCE REQUIREMENTS (flow-next's own work-loop quality gates / Stop hooks) - this is the home of "tests must pass"
  - L4 pre-merge: CI required checks running the SAME verify command - the only gate agents cannot route around locally
  Pass = tests gated at L3/L4 (the requirement). L1/L2 fast file/staged-scoped enforcement with real content = a strength; its ABSENCE is a headroom warn, never a pass-blocker - "no git hooks but hardened CI + verify command" is a legitimate pass (eval caught the earlier conjunctive wording failing the exact repo the example blesses). Flags: stub hook content (`.husky/` echo scaffold, whitespace-only pre-commit config) = ❌ with evidence quoted; test-suite/E2E/coverage in pre-commit = ⚠️ "heavyweight hook - known agent bypass/stall risk; move tests to verify + CI"; git hooks in an agent-committing repo without bypass hardening (deny on `git commit --no-verify` / PreToolUse guard) = ⚠️ "advisory only for agents". L3/L4 divergence from L1/L2 tooling (different tools/flags per layer) is itself a finding - one verify command is the single source of truth.
- **TS5 coverage thresholds:** pass requires an ENFORCED threshold (fail_under / --cov-fail-under / branches: in config), not coverage tooling existence - a threshold of 0 is the same stub pattern
- **SV3 type-checking depth:** grade strictness substance (strict flags enumerated; bounded `: any`-ratio probe on TS), not "TypeScript exists" - type strictness is a blocking dimension in ceiling-scored rubrics (agents lean on types as their primary in-context verifier)
- **SV5/SV6, BS2/BS3:** the named script must exist in the manifest AND execute (Phase 2 extension); a `lint` script that crashes is ❌ with the error
- **DC2 instruction-file quality (claude-md-scout rubric deepened):** grade on substance tells - repo-specific nouns, >=3 fenced copy-paste-runnable commands in the top ~50 lines, commands appear verbatim in manifests, 1-3 real code snippets, three-tier boundaries with concrete paths, don'ts paired with dos; template tells (generic personas, restated universal conventions, full directory listings, unedited /init scaffold order, placeholders, zero fenced blocks) are DEDUCTIONS. Length band 30-150 lines; >300 is a REVIEW TRIGGER routing to prune-to-pointers/nested-files remediation (catalog 10/11), NEVER an automatic quality deduction - substance tells rule (eval: a 659-line file passing every substance tell earned the prune recommendation, not a downgrade; a 255-line file was 54% generic preset boilerplate only the tells could see). Also flag instruction-file SPRAWL: multiple root-level instruction files (CLAUDE.md + AGENTS.md + AGENT.md + GEMINI.md) beyond the canonical-plus-bridge pattern. ETH Zurich (arXiv 2602.11988): LLM-generated instruction files reduce task success ~3% at +20% cost while hand-curated add ~+4% - so prime downgrades generated-looking files and its own remediation NEVER bulk-generates a full CLAUDE.md (offers a short, commands-first skeleton the human fills, or an interview-derived augment)
- **DC2 execute check (host, Phase 2):** run 1-2 commands quoted in the agent file; a CLAUDE.md whose stated test command fails is worse than none (feeds gate G3). Extraction is HOST-AGENT work (read fenced blocks + inline backticks, take each command's leading token, resolve against tracked files / manifest scripts / PATH); **zero commands extracted from a file that HAS fenced blocks = an extraction-failure flag, never a vacuous G3 pass or a stub grade** (eval: naive extraction found 1 of 15+ commands in a best-in-class file and 0 of 26 in another - both would have silently gutted the gate)
- **DE1 .env.example:** diff declared vars against env reads in SOURCE (`process.env`, `os.environ`, `getenv`); >~30% undeclared = "stale template" ⚠️. Eval-hardened scoping: iterate WORKSPACE MEMBER dirs, not root-only (per-package .env.example is the correct monorepo pattern - a root-only probe false-negatived 4 real files and would have emitted a visibly wrong top-5 action); scan only source extensions in the git index, excluding tests/fixtures/eval corpora/docs snippets/node_modules (two eval repos' "stale" evidence was markdown runbook snippets and teaching content inside a French test fixture - the quote-your-evidence rule makes such hits self-evidently invalid); filter well-known platform/CI vars; downgrade "stale template" to "template does not mirror documented config" when the vars are documented in a config doc
- **DE4 setup script:** must chain real stages (install AND migrate/seed keywords found, or it executes); a setup.sh that only prints instructions is ❌
- **DE5 devcontainer:** empty json with no features/postCreateCommand = "checkbox artifact" ⚠️, not ✅
- **Content the agent could discover itself** (directory trees, standard conventions restated) counts as negative signal in DC2, never neutral

Substance judgment stays skill/scout work anchored to quoted file content and executed evidence (host agent IS the judge, per repo architecture rules) - no new regex engines in flowctl. Fabrication guard: every substance verdict quotes its evidence (file line or command output); no evidence, no verdict.

### Agent observability & drivability (new check group; extends ladder tier 3)

The ladder's tier 3 ("runs") splits into sub-signals: runs / OBSERVABLE (the agent can see what the running app did) / DRIVABLE (the agent can exercise it end-to-end). Anthropic's best-practices framing: the closed verify loop is "the single most impactful tip" - give the agent a check it can run and have it show evidence. This is agent readiness (fix-offered), distinct from report-only production observability (Pillar 6). Criteria (14 of 16 are pure bounded greps with quoted evidence; the 2 executed probes REUSE the tier-3 boot probe and the existing CLI --help pattern - no new execution budget):

- **AO agent-readable runtime feedback (scored):** dev-server logs readable as files (log path/recipe in agent file, agent-tail-style `tmp/logs/` layout; token-cheaper than MCP tool schemas); browser console reachable (chrome-devtools/Playwright MCP entry or console-capture script); parseable ready line + deterministic port (from the boot probe); dev request-logging middleware; documented DEBUG/LOG_LEVEL escalation
- **DR drivability (scored):** seeded/demo data one-command (merges with catalog item 3); DOCUMENTED dev login path / test user (env-gated - every drivability path dies at the login wall without it); curl-able API + health endpoint (OpenAPI/route-list + dev token); stable selectors (data-testid/roles/labels counts - the substrate browser agents drive via the a11y tree); browser-automation harness present (Playwright/Cypress config, screenshot command documented); agent-first CLI for service-shaped apps (--json, stdout=data/stderr=errors, stable exit codes, --dry-run on mutations); framework dev-MCP where the stack ships one (Next 16+ next-devtools-mcp - stack-gated via stacks.md rows)
- **TO test-time observability (scored):** e2e failure artifacts configured (trace on-first-retry, screenshot only-on-failure, video retain-on-failure); app logs captured on e2e failure (playwright webServer stdout 'pipe'); real reporter with assertion diffs (extends catalog item 5); flaky-quarantine visible not hidden (retry config + annotated skips with tickets; >~5-10% quarantined = systemic flag)
- **DT dev-loop telemetry (informational only, like DC8/DE7):** dev-mode OTel wiring (jaeger/collector in compose, OTEL_ vars in .env.example); trace-query surface for agents (otel-mcp/grafana-mcp or documented "what happened when I hit /checkout" recipe). Adoption is genuinely early - never scored, suggestion line only.

Stack- and tier-gating per the pattern discipline: ladder-capped stacks (tier 1-2 ceilings) get N/A whitelist entries for AO/DR web criteria; CLI-shaped and library repos swap DR web rows for the CLI row.

**QA-readiness linkage (`/flow-next:qa` / `pipeline.qa`):** the ladder + DR group double as the readiness gate for flow-next's own live-app QA stage - `/flow-next:qa` is forbidden from passing by reading source and needs exactly what DR measures (seeded data, documented dev login, drivable surface, readable evidence). The report emits an explicit QA-readiness line: tier 3 + DR core passing -> "QA-ready: consider `/flow-next:qa` / enabling `pipeline.qa`" (the recommendation to enable the stage fires ONLY here); anything less -> "QA stage would fail here: <the missing DR items>" so teams fix the prerequisites before switching the stage on. Shape/tier-capped repos (library/plugin, tier-2-ceiling stacks) state "QA stage not applicable to this shape" rather than a gap.

### Harness & Permissions (new check group - the front-door pillar no rubric has)

Prime today assesses the codebase but not the checked-in HARNESS configuration that makes agents safe and effective in it. Existing rubrics only existence-check a file or two; the substance/cross-reference angle is wide open. Two governing principles:

1. **Score the function, not the file.** Five function classes - (a) instructions, (b) permission/sandbox posture, (c) lifecycle hooks, (d) MCP config, (e) cloud-agent/env bootstrap - and each harness implements a different subset. Detect ACTIVE harnesses first (config dirs present + commit recency); grade only those, map each criterion to the harness's native mechanism, N/A otherwise (never fail a Codex-only repo for missing `.claude/hooks`; Cursor's "permissions" are access-scoping via `.cursorignore`; Codex posture = config.toml + trust). One deep harness + an AGENTS.md bridge outscores five shallow config stubs.
2. **Zero harness config is today's normal** - the group reads as headroom, not shame. EXCEPT two P0 findings regardless of scoring: inline secrets in MCP config, and suspicious hook content.

Criteria (HP set; core scored subset marked, rest informational at first):

- **HP1 cross-harness instruction bridge (scored):** one canonical instruction file; the other harness's file is an `@AGENTS.md` import/symlink/thin adapter, NOT a diverged duplicate (Claude Code does not read AGENTS.md natively - the import IS the bridge); diverged duplicates = fail with diff summary. **Resolve the SYMLINK GRAPH before diffing and report it whole** (eval: a repo had nested package files symlinked INTO the root AGENTS.md - naive pair-diffing miscounts, and unifying the root pair without the graph silently breaks the package guide the symlink serves). Eval validation: both real diverged-duplicate findings were true positives and UNDERSTATED (423 lines two-way drift; 866 changed lines) - this check is a keeper.
- **HP2 project permissions file present + parses (scored):** `.claude/settings.json` (or `.codex/config.toml` / `opencode.json` equivalents) valid with a permissions block
- **HP3 verify-loop allowlist (informational):** the repo's own Phase-2-verified test/lint/build commands are pre-approved (allow rules), so agents run the loop promptless; cross-referenced against manifests - a dangling allow rule naming a nonexistent command = stale-config warn (HP4)
- **HP5 secrets deny rules (scored):** `.env` family + repo-specific secret paths deny-read in each active harness (`permissions.deny Read(...)`; Cursor parity: `.cursorignore` carries `.env*`, keys, secrets dirs)
- **HP6 destructive/network guardrails (informational):** curl/wget/rm -rf denied or sandbox-gated; push-class ops on ask
- **HP7 hook substance + safety screen (scored):** enumerate configured hooks; READ each including the hook COMMAND STRINGS (bounded, never execute during assessment) - capturing `hooks: true` without content makes HP7/HP8 unfireable (eval: BOTH real repos with hooks wired whole-project type-aware lint with `--fix` to PostToolUse on every edit - the HP8 warn verbatim - and the boolean-only capture missed both); classify real gate / stub / suspicious (network calls, credential paths, obfuscation). Stub = fail-with-quote; suspicious = P0 security finding - committed hook config is an RCE vector (CVE-2025-59536 class); heavy whole-project checks wired to PostToolUse (fires per edit) = perf warn (HP8)
- **HP9 MCP secrets by indirection (scored):** `.mcp.json`-family values use `${VAR}` env indirection; any inline literal secret shape = FAIL quoting the KEY NAME only, never the value, with "rotate it - it is already in git history"
- **HP10-11 MCP hygiene (informational):** stdio commands resolve, versions pinned not @latest, server count sane (tool bloat warn ~10+); each server has a when-to-use line in the agent file
- **HP12 personal-scope files not committed (scored):** `.claude/settings.local.json`, `CLAUDE.local.md`, `.env` absent from the git index (eval: hit on 2 of 5 real repos, one twice). **When a local-scope file IS committed, its CONTENT feeds the HP3/HP5/HP7 posture assessment** - eval found a committed settings.local.json carrying the repo's real 50-rule allow surface while settings.json was empty; reading only settings.json misstates the posture. Related: HP5's "repo-specific secret paths" gains a discovery mechanism - derive deny-rule candidates from the repo's own .gitignore secret-shaped entries (eval: a gitignore explicitly named a plaintext-credentials file that sat agent-readable in the worktree)
- **HP13-16 (informational):** cloud-agent env config parity with verified install commands (ties to DE5/catalog 9); sandbox/isolation posture ladder (none < deny-rules < sandbox/network-allowlist < devcontainer firewall; Windows hosts: N/A never fail); path-scoped rule hygiene (dangling `paths:`/`applyTo:` globs matching zero tracked files = stale-rule warn); PR-template agent-evidence section

Remediation safety rules (hard, into remediation.md): deny/ask rules are the ONLY permissions artifact safe to scaffold (they apply without workspace trust and only restrict; model on anthropics/claude-code examples/settings). Allow rules: propose only entries derived from commands prime itself executed successfully in Phase 2, and state that project allow rules are inert until the workspace-trust dialog is accepted. Never scaffold: defaultMode changes, bypassPermissions, danger-full-access, approval_policy never, additionalDirectories, HTTP hooks, MCP server installs. Hooks offered = built from verified commands + read-back gated + exercised in the same pass. Secret findings: key name only, never the value, always "rotate".

### Feedback-latency + hygiene criteria (from the 13-rubric gap-diff)

Net-new checks the gap-diff surfaced (deduped against everything above; taxonomy per prime's fix-offered vs report-only split):

- **Docs freshness vs code churn (scored, DC extension):** `git log -1 --format=%ct` on CLAUDE.md/README/docs vs src churn; instruction files untouched for months while src churned = drift flag (the failure mode every 2026 source independently names; complements DC2's point-in-time quality). Fix = targeted refresh offer, never bulk regeneration.
- **Large-file/legibility metrics generalized (scored):** LEG4's pathology inventory promoted from the huge/legacy playbook to ALL repos - p50/max file LOC + top-N offenders via scc (already a dependency); fix = report offenders + offer a max-lines lint rule.
- **Feedback latency MEASURED (report lines, not new probes):** local suite wall time per Planning resolution 3 (time only what already executed) + CI wall time derived locally from `gh run list --limit 20 --json startedAt,updatedAt,status,conclusion,headBranch` on completed default-branch runs (there is NO durationMs field in the gh JSON surface - compute updatedAt minus startedAt); >~10 min median caps agent iteration. The metric the weighted rubrics put their heaviest points on; build caching config (turbo/nx/actions-cache) reported alongside.
- **CI actually gates (scored):** workflows contain test AND lint steps AND **gate-relevant TRIGGERS** (pull_request / push on the default branch) - step content alone is insufficient (eval: a repo whose ONLY workflow was a tag-triggered deploy carrying test+lint steps passed the naive wording while agents could merge anything; the trigger gap was that repo's single largest finding). When `gh` is authed, corroborate with required-status/branch-protection. Two adjacent smells: a CI "lint" step that MUTATES (`--write`/`--fix`) instead of checking can never fail - flag it; external deploy platforms (vercel.json/.netlify) are INFERRED as a compile gate, so the recommendation becomes "add test-gating CI (compile gate already external)" and tags=0 is suppressed as a signal for continuously-deployed web repos. Coverage-threshold enforcement rides TS5 above.
- **VCS CLI available + authed (INFORMATIONAL host-environment line, excluded from ALL repo scores - a machine property must never make the same repo score differently per assessor):** `command -v gh && gh auth status` - prime's own Pillar 7 probes and every downstream PR skill silently degrade without it; today it is assumed silently. Reported as a host-prerequisite line in the report header.
- **Local secrets gate (scored, SV4-content extension):** gitleaks/detect-secrets/trufflehog in the commit gate or CI - agents generate and commit fast; server-side scanning (SE2) fires after the leak. Push protection status reported alongside SE2 (report-only).
- **Destructive-script scan (scored, severity-classed - eval-hardened):** bounded rg for recursive-delete/`--force`/`push -f`/db-drop patterns in manifest scripts, Makefiles, and scripts/. Findings classify by CONTEXT AND TARGET, not pattern alone (eval: 5 raw hits on one repo = 1 echo string-literal + 4 benign self-managed regenerate-dir deletions; blanket never/ask naming would generate agent-file noise in every repo with installers): string-literal/comment/doc-snippet context = dropped; repo-internal dir the same script regenerates = informational, feeding a LEG7 never-edit line ("codex/ is generated - never hand-edit") rather than a destructive warning; $HOME/env-derived bounded path = ask-tier mention; unbounded or parameterized target = P1 and named in the never/ask tiers (an agent innocently running `npm run clean` that nukes a sibling dir stays the motivating case).
- **API contract presence (conditional, scored):** when an HTTP framework is detected - OpenAPI/GraphQL schema/proto files as a machine-verifiable target agents can diff against; fix = suggest generation from routes.
- **Module-boundary enforcement (informational, size-tiered):** import-linter/dependency-cruiser/eslint-boundaries/Nx tags/ArchUnit config - turns DC6's prose architecture into an executable check; suggest only above the size tiers where it pays.
- **Report-only additions:** dependency/runtime currency (runtime major vs EOL table; ancient stacks fall outside model training distribution), test isolation/parallel safety (parallel agents + worktrees are flow-next's own model), flaky-test signals (retry config, re-run rate), conditional LLM-eval harness (only when an LLM SDK is in deps: evals/ dir or promptfoo/braintrust config - repos shipping LLM features need a verify loop for prompts like tests for code).

### Report shapes per classification (playbooks.md)

**Greenfield:** the 48-criterion scorecard is suppressed; emit an ordered **bootstrap plan** (~8-12 stage-appropriate items): hand-curated agent-file seed (short, commands-first), STRATEGY.md first (`/flow-next:strategy` exists), stack decision recorded with exact versions, verify loop before feature 1 (test + smoke), smallest CI (install+lint+test), hygiene files (.gitignore, lockfile, .env.example, .editorconfig - never LICENSE per guardrails), first spec = first vertical slice with explicit non-goals. Pillars that are premature get **recorded-deferral N/A** ("no observability yet - deferred until first deploy"): a documented deferral beats BOTH a silent gap AND a stub file - the stub is the rot generator this spec exists to kill. Anti-pattern rules: prime never emits stub artifacts that would pass its own checks (anything scaffolded is exercised in the same pass); no big-bang scaffolding; no heavyweight spec ceremony on an empty repo.

**Standard single repo (small/medium):** current report, upgraded with the verdict headline, substance criteria, and size-tier recommendations. Below 400K LOC do NOT recommend LSP/index tooling (measured net-negative); recommend orientation map + loops instead.

**Monorepo (exemplar: modern TS pnpm/Nx/Turbo):** usually already legible - the build graph IS the map. Playbook: (1) verify per-package commands work from each package dir and are file-scoped (`pnpm --filter <pkg> test`, affected-only where Nx/Turbo exists); (2) nested per-package CLAUDE.md/AGENTS.md when >2-3 distinct subsystems or root file >~200 lines (nearest-wins per the AGENTS.md spec; reference point: OpenAI's Codex repo carries 88); (3) scoping config - read-deny for `dist/`/generated, sparse worktrees for subagents, launch-from-package-dir guidance; (4) if Nx present, recommend its agent wiring (graph + affected-only as the feedback loop); (5) verify the lockfile/task-runner single-source rule (one package manager). Score per existing pillars; monorepo N/A handling (BS6) already exists.

**Huge and/or legacy:** legibility-first block AHEAD of pillar detail. The playbook is a set of NAMED GENERIC PATTERNS - skill logic and report prose stay stack-agnostic; each pattern's concrete instantiation for a given stack lives ONLY in that stack's `stacks.md` row. (Delphi at 15M LoC is used below as the worked exemplar of each pattern, not as skill logic.)

- **LEG1 Honest tier + headless-feedback-loop wrapper:** name the operability tier from evidence; the top recommendation at tier 1 is always "one-command headless build per project/module" so the agent has a compile feedback loop. *(Delphi: `Build.cmd` wrapping `rsvars.bat && msbuild X.dproj`; one paid seat legally covers an unattended build server. .NET Framework: `msbuild /t:Build /restore`, never `dotnet build`.)*
- **LEG2 Toolchain/license reality check:** detect when build or code-intelligence tooling is bound to a licensed IDE/machine or an unavailable platform; report what CAN run headlessly on which host, never fabricate a pass for what cannot run here. *(Delphi: DelphiLSP license-bound; Windows-only builds unprobeable from macOS/WSL hosts -> "not probed on this host".)*
- **LEG3 No-LSP/no-map honesty + substitute navigation artifacts:** when the stack matrix says code intelligence / `/flow-next:map` is not practical, say so and recommend the substitute class: (a) a generated dependency-graph artifact, checked in or regenerable - deterministic dependency maps are prerequisites, not nice-to-haves (Microsoft COBOL-agents lesson); (b) a hand-written orientation map (top-level dirs, entrypoints, module one-liners); (c) static analysis as the proxy verifier where tests are absent. *(Delphi: DUDS/MMX/Pascal Analyzer uses-graph; entrypoints from the .dpr; SonarDelphi/FixInsightCL/SARIF.)*
- **LEG4 Pathology inventory as first-class findings:** top-N largest files (agents without symbol tooling read whole files - token burn), generated-file globs, binary/opaque artifacts, vendored trees - all excluded from agent-readable scope in the agent file. *(Delphi: 50K-line units, .res/.dcu, binary .dfm.)*
- **LEG5 Encoding/codepage sweep (P1 finding):** BOM/charset sniff on a sample per extension; legacy sources are often ANSI/Windows-1252 or UTF-16, and coding agents have documented corruption bugs on non-UTF-8 files (anthropics/claude-code #7134, #28523, #28316, #50717). Recommend: deliberate one-time normalization commit, or an encoding-guard hook, or a never-edit list for affected files. *(Applies equally to old C#/VB dumps and UTF-16LE SQL scripts.)*
- **LEG6 Atomic file-pair rule (ALL SHAPES, like LEG4):** detect designer/generator paired files where editing one side without the other corrupts the project, and require a pairing rule in the agent file. *(Instances: Delphi .dfm+.pas, WinForms .Designer.cs+.cs, Qt .ui+, Xcode storyboard+swift, protobuf generated+source - and modern equivalents: a dual-copy invariant file pair (eval: scripts/flowctl.py <-> .flow/bin/flowctl.py, documented only in the maintainer's private memory - a real trap for any other contributor's agent).)*
- **LEG7 Tool-managed never-edit list (ALL SHAPES, like LEG4):** files/dirs an IDE or toolchain rewrites, regenerates, or that are semantically opaque get an explicit never-touch list in the agent file. *(Instances: .dproj/.dsk/.identcache/__history, xcodeproj internals, .sln user files, .idea/ - and modern equivalents: a script-regenerated mirror dir ("codex/ is generated by sync-codex.sh - never hand-edit"), tracked legacy snapshots. Eval promoted LEG4/6/7 from the legacy playbook to all shapes: both patterns demonstrably fired on a modern plugin repo.)*
- **LEG8 Module carve-outs as the unit of agent-workability (strangler-fig):** name which subsystems are safe agent territory vs frozen; a carve-out with its own build + tests is what "agent-ready" means at multi-M LOC - never the whole tree. Where the ecosystem's honest answer is migration tooling rather than in-place agent work (see matrix rows), say that too.
- **LEG9 Characterization/snapshot tests as the verify strategy** where unit tests are absent and behavior must be preserved during modification.

**Constellation-member (exemplar: 99 repos):** per-repo priming STAYS VALID - every repo still gets its own assessment and fixes; the constellation block is ADDITIVE guidance on top. Recommend the **home-base pattern** (industry-consensus "repo-of-repos"/"virtual monorepo"): parent dir (optionally a tiny git repo) with siblings as plain gitignored checkouts - NOT submodules (update friction, worktree incompatibility, agents forget init; exception: deliberate version-pinning); the highest-leverage artifact is a **repo manifest with one-line purposes** (`repos.yaml`/md) that a LEAN parent CLAUDE.md points at (manifest = single source of truth, never duplicate the list). Home-base contents: lean parent instruction file (workspace map, cross-repo change workflow, git-directory-safety rule, docs-update-as-DoD), the manifest, clone-all/status-all/test-all scripts (mani/gita/vcstool or plain scripts; mise `monorepo_root` for task namespace), constellation docker-compose, ports + env bootstrap map, `_plans/` for cross-repo plans. Platform caveat stated in the report: Codex ignores an AGENTS.md above the git root (openai/codex#15683), so per-repo files must link BACK to the home base explicitly; Claude Code `--add-dir` solves access, not knowledge. Cross-repo change choreography documented in the home base: contract-first ordering (schema -> provider -> consumers), one repo per worker, linked PRs, merge order libraries-before-consumers, contract tests at boundaries. Scale ladder: manifest + home base at 2-30 repos; dependency/release-ordering registry at 20+ (Mabl's 79-repo registry cut context-drift failures ~40% -> <5%); recommend TRUE monorepo consolidation only when cross-repo changes dominate (>~30-50% of features touch multiple repos) and no isolation constraint exists - the home base is the middle option capturing most of the benefit at near-zero migration cost. Prime running in the home-base dir itself assesses the constellation layer (manifest present? compose boots? parent file lean + pointing?).

**Lightweight product-family variant (eval-driven):** a 2-3 repo family (code repo + docs site + sister product) whose coordination need is docs currency, not service composition, must NOT receive home-base/compose/ports boilerplate - the microservice-shaped playbook is oversized there. The right emission is the R15 "Repo context" block naming the siblings plus a docs-update-as-DoD line in the agent file. playbooks.md names both variants and the selector (service composition detected -> full home base; prose/docs coupling only -> light variant).

## Edge Cases & Constraints
<!-- scope: technical -->

- **Boundedness is load-bearing:** every probe is deterministic and capped - `git ls-files` counts, `find -maxdepth`, config presence, ONE sampled grep probe, `scc`/`tokei` (never `cloc`, never exhaustive reads); execution probes get timeouts (build ~5 min, test-list ~2 min, boot ~60s) and size caps on file reads (giant generated files). No full-repo LLM crawl - that is the exact case that breaks.
- **Non-mutating execution policy (plan-review round 3 - CRITICAL; supersedes the looser trust-surface wording):** assessment must never mutate the assessed repo. Rules: (1) STATIC resolution for every documented command (does it exist in manifests/PATH) - always safe; (2) EXECUTION only for allowlisted evidence classes: lint/typecheck in CHECK mode, test discovery/list (dry-run), bounded build, `--help`/`--dry-run`, and the gated tier-3 boot probe; (3) formatters only via check mode (never `--write`/`--fix` against the worktree); (4) setup/migrate/seed/deploy/destructive/network commands are NEVER executed during assessment - static + cross-reference evidence only (DE4 grades by content, not execution); (5) BS3 dev-command evidence comes from the tier-3 boot probe, never a second long-lived server run; (6) pre/post `git status --porcelain` snapshot around every executed probe - unexpected tracked changes INVALIDATE that evidence and become a finding in their own right; (7) G3 distinguishes "resolves" (checked for ALL quoted commands) from "safe sampled execution" (run only for allowlisted classes). Windows-only builds (Delphi, .NET Framework) that cannot run on the current host are recorded "not probed on this host", never fabricated ✅ - the unverified-counts-as-fail rule extends to every executed criterion.
- **Classification is heuristic:** three-tier confidence, evidence always printed, low-confidence calls confirmed interactively; autonomous/`--report-only` runs state the assumption. Misclassification must degrade gracefully - a monorepo misread as single repo still gets a correct base report, only the playbook block is off.
- **Fabrication guard:** every substance verdict quotes its evidence (file line or command output). A criterion with no evidence is NOT ASSESSED, never guessed. Existing scout-failure and five-state scoring rules (N/A whitelist, ⚠️ unverified) stay authoritative; classification only ADDS N/A entries via the documented whitelist, the model still may not invent N/A.
- **Do not regress the existing scan:** all 48 criteria remain; substance upgrades tighten pass conditions, never remove checks. Existing guardrails hold (never modify code files, never commit, never delete, never LICENSE, pillar 6-8 report-only, DC7/DE7/DC8 handling, glossary read-back).
- **Size-tiering is bidirectional:** below 400K LOC do NOT recommend LSP/index/code-intel tooling (measured net-negative); above it, DO. Over-recommending equals under-recommending as a failure.
- **Stack honesty over aspiration:** where the matrix says "none practical" (Delphi LSP, PL/SQL, VB6 maps), prime must say so and recommend the realistic alternative (dependency-graph artifact, hand-written map, static analysis as proxy verifier) - never suggest `/flow-next:map` on a stack clawpatch does not parse (extend the DE7 suggestion gate with the stack check).
- **Never scaffold what would pass prime's own checks unexercised:** every remediation artifact prime creates is exercised in the same pass (command run, hook invoked) or explicitly marked unverified. LLM-bulk-generated CLAUDE.md is forbidden as a fix (measured harm); offer skeleton + human fill or interview-derived augment.
- **Encoding hazards:** the encoding sweep must not itself corrupt anything (read-only probes); recommendations only.
- **Harness config is an attack surface, not just an asset:** committed hooks/settings execute on teammates' machines at workspace trust (CVE-2025-59536 class). Prime READS hook scripts during assessment, never executes them; suspicious content is a P0 finding. Secret findings quote the key name only, never the value, and always instruct rotation. Never report "allow rules present" as "prompts eliminated" - project allow rules are inert until workspace trust is accepted; remediation text must say so.
- **Harness fairness:** grade only ACTIVE harnesses (config presence + commit recency); map criteria to each harness's native mechanism or N/A. Claude sandbox unavailable on Windows hosts = N/A, never fail.
- **Provider-free stays:** prime requires no API keys/providers; `scc`/`tokei`/`rg` are optional accelerators with fallbacks; clawpatch stays optional (DE7).
- **`--classify-only` must stay cheap** (<~10s on a huge repo): it is the portfolio-triage entry point for 100+ repos.
- **Home-base assessment mode** (prime run in a parent dir that is not itself a project repo): detect via manifest + sibling checkouts; assess the constellation layer instead of erroring on "no manifest found".

## Acceptance Criteria
<!-- scope: both -->

- **R1:** Prime classifies every repo on five axes (lifecycle greenfield/hybrid/brownfield; topology as TWO independent bits - monorepo and constellation-member with the workspace dampener; size band with the tool-managed/vendored/fixture/duplicate exclusions; MANIFEST-GATED stack set; multi-valued delivery shape) using only bounded deterministic probes, prints the classification with evidence and per-axis confidence fields, and confirms low-confidence calls interactively (states the assumption in non-interactive runs). Regression: the eval fixture set (workspace parent, tier-a/tier-b constellations, greenfield, greenfield-x-constellation) plus the 5 real-repo expectations classify correctly.

## Decision Context
<!-- scope: both -->

### Motivation
<!-- scope: business -->
At portfolio scale the binary existence-check produces false "ready" signals and no actionable guidance, and leads cannot hand-verify hundreds of repos across many stacks. Encoding a senior's first-look judgment - size, legibility, topology, operability tier, single most-valuable next action - makes prime trustworthy at scale and directly unblocks portfolio-scale multi-repo, multi-stack rollouts (the current bottleneck). `--classify-only` gives the portfolio lead a cheap triage sweep across 100+ repos before any deep assessment.

### Implementation Tradeoffs
<!-- scope: technical -->
Decisions made in lieu of the interview (flag any to revisit):

- **Default behavior, not a new mode.** Keeping the existence-only path reachable would preserve the failure mode fn-92 retires. Only `--classify-only` is added. Revisit if a "fast legacy-compatible run" proves necessary.
- **Operability ladder over boot-probe-by-default.** Tier 3 probes run only on cheap ready-signals with timeouts; the ladder makes legacy honest instead of failed. Alternative considered (opt-in `--boot` flag) rejected as another knob; the ready-signal gate bounds risk equally.
- **Hard gates cap at Level 2** rather than zeroing the score: preserves score comparability while killing false Level 5s. The gate list is deliberately tiny (3) to stay uncontroversial.
- **Classification host-inline, no new scout:** it is deterministic bash, not judgment; a scout adds latency and a failure mode for no gain. Scout dispatch CONSUMES the classification instead.
- **Four reference files** (classification/playbooks/stacks/harness) over inflating workflow.md: mirrors pillars/remediation precedent; keeps always-loaded prose small (fn-82 discipline).
- **stacks.md as maintained opinion table** over auto-detection cleverness: rows are explicit, reviewable, and correctable per portfolio experience; unknown stacks degrade generically.
- **One deterministic flowctl emitter, no judgment in flowctl** (AMENDED at planning - supersedes the earlier no-plumbing default): `prime classify --json` ships per Planning resolution 19 because the probes are mechanical (the flowctl carve-out), the eval needs a deterministic oracle, and portfolio sweeps need a scriptable surface. Judgment (shape reasoning, confidence, asks, playbooks) stays in the skill.
- **Greenfield suppresses the scorecard** rather than scoring with N/As: a 10-item bootstrap plan is the honest report for an empty repo; a scored husk invites Goodharting the checklist from day one.
- **Pattern-level playbooks, stack data in rows:** an earlier draft embedded Delphi specifics in the playbook prose; reworked so skill logic references only generic patterns (LEG1-LEG9) and stacks.md rows carry the instantiations (Gordon's steer 2026-07-11: never too specific to one stack). Adding a stack = adding a row.
- **Hooks demoted from requirement to layer choice** (Gordon's steer 2026-07-11: "not 100% sure about husky hooks for tests - acceptance requirements instead"): wave-2 evidence (the #40117 bypass incident, Copilot's terminal hook failures, Factory still requiring hooks via existence checks) supports making SV4 layer-agnostic and moving "tests must pass" to acceptance requirements + verify + CI. flow-next's own work loop already owns that gate; a test-running pre-commit hook duplicates it with a worse failure mode.
- **AO/DR as ladder sub-signals, not a tenth pillar of their own:** lighter, keeps the "cheapest move up one tier" framing, and reuses the existing boot probe for all executed evidence. DT (dev-loop telemetry) stays informational - adoption is genuinely early and scoring it would punish the mainstream.
- **HP scored-core vs informational split:** zero harness config is today's normal, so most HP rows start informational (headroom framing); the scored core is the safety-critical + highest-signal subset. Revisit the split as the ecosystem matures.
- **Gap-diff triage recorded:** 40 candidates from 13 rubrics; high/med-leverage items adopted above, low-leverage exclusions listed in Architecture so planning does not relitigate them.
- **Eval-first pass done pre-planning** (Gordon's steer: test on real repos + a fake multi-repo root): 5 real repos + 3 fixtures + 4 judge agents. Produced the Axis 5 delivery-shape addition, the two-bit topology, the exclusion lists, the trigger-aware CI check, and a dozen probe hardenings - all folded in above with "eval:" evidence markers. Real findings shipped as side effects (diverged instruction bridges and committed settings.local.json on the maintainer's own repos). The prototype harness lives in the session scratchpad; planning re-creates it as a proper test asset (R19) rather than inheriting the prototype.
- **Ask, but bounded and durable:** classification cannot always infer org context (siblings elsewhere, licenses, frozen modules). Chosen shape: one AskUserQuestion budget per run, evidence-stated questions, answers recorded into the agent file's "Repo context" block so asks converge to zero over runs; autonomous runs surface an Unresolved-questions list instead of blocking. Alternative (always-silent assumptions) rejected: wrong-topology playbooks are costlier than one good question.
- Thresholds (~25 commits, ~30 files, 100K/400K/2M LOC, 20K files, ~200/~300 line instruction-file bands, 30-50% cross-repo change rate, 20+ repos for a registry) are research-anchored starting opinions living in classification.md/playbooks.md - tune with portfolio data, not in flowctl code.
