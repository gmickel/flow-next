# fn-89-tracker-sync-lifecycle-dispatches-off Tracker-sync lifecycle dispatches off the critical path (background subagents)

## Goal & Context

Every tracker touchpoint today runs INLINE in the host skill: the work loop waits on Linear/GitHub API round-trips between tasks, and — worse for context — the tracker machinery pollutes the host's window. Evidence from the fn-88 run (2026-07-05): each MCP `save_issue` echoed the entire issue body (~10k tokens) back into the HOST context, twice; a body reconcile loads ~13–17k tokens of merge-doctrine refs into the host. The wall-clock cost is seconds per touchpoint; the context cost is the real tax.

The fix is NOT "parallelize everything" — it is **off-the-critical-path + context-isolated**: lifecycle event dispatches execute in a background tracker-runner subagent (own context absorbs the skill refs and MCP echoes; host gets one line back), overlapping the long-pole work (worker runs are minutes; tracker calls are seconds). Two hazards shape the design: per-spec sync-state races (merge-base/lastSyncedAt snapshots must not interleave) and the Phase-5/end-of-run `sync check` audit racing an in-flight dispatch (false MISSING → duplicate retro-fire).

**Scope split:** ceremonies and interactive genuine-conflict resolution STAY inline (they need `AskUserQuestion`; subagents can't block on questions — the documented reason tracker-sync is an inline skill today). Only the event-tagged lifecycle dispatches (capture/interview/plan reconcile, work.firstClaim push, work.done/resolvePr/completionReview/qa comments, makePr reconcile) become forkable — they never ask under the autonomy policy anyway (genuine conflicts queue via `sync defer`).

## Overview & Approach

The touchpoint gates already exist and are byte-identical in shape (fn-52 shared gating predicate) — this spec changes only **where** the tracker-sync body executes once a gate fires: from inline in the host to a dispatched `tracker-runner` subagent. Every anchor below is a current file:line the implementation edits in place.

1. **Tracker-runner subagent (`agents/tracker-runner.md`, NEW).** `model: sonnet` (feasibility "Runner tier": sonnet-class sufficient for both shapes — its own routing win, not the conductor's model), `disallowedTools: Task` **only** (KEEPS `Write`/`Edit` — the reconcile writeback in `references/body-merge.md` Step 5 writes the merged flow body to `.flow/specs/<id>.md` via Write/Edit, so a Write/Edit-blacklisted runner cannot execute a state-shaped reconcile — plus `Bash` for flowctl/curl and inherits `Read`/`Grep`/`Glob`; MCP loaded in-fork via `ToolSearch` per feasibility T2; precedent: `agents/pr-comment-resolver.md`, a writing subagent that blacklists `Task` only). Contract inputs: `OPERATION` / `SPEC_ID` / `EVENT` / `FLOWCTL` / `DISPATCH: forked` + the autonomy markers (`FLOW_RALPH` / `FLOW_AUTONOMOUS` / `REVIEW_RECEIPT_PATH`). It runs the EXISTING `flow-next-tracker-sync` skill body for that ONE op (no second implementation — R7) and returns **exactly one outcome line**: `<status> <spec-id> <note>`, `status` = the FULL tracker receipt enum `pushed|pulled|merged|updated|diverged|queued|errored|noop` (perEvent leaves can be `pull` and a reconcile can receipt `diverged` — a narrower outcome enum would force the runner to lie about valid outcomes). Fleet convention: every dispatched subagent has an `agents/*.md` contract; the `.md → .toml` loop in `sync-codex.sh:1367-1372` auto-generates `codex/agents/tracker-runner.toml`.

2. **Dispatch shape at each touchpoint.** The gate's dispatch line changes from an inline `skill: flow-next-tracker-sync (operation: …, event: …)` to a host `Task flow-next:tracker-runner` call carrying the same `operation`/`event` plus `DISPATCH: forked`. The dispatch string introduces a `Task flow-next:` token that will appear in ~8 mirror skill markdown files across .1/.2/.3, so `sync-codex.sh` needs a **single GLOBAL rewrite** — a `find "$CODEX_DIR/skills" -name '*.md' -exec sed …` sweep mapping `Task flow-next:tracker-runner` → the `tracker_runner` agent-role form — landed once in .1 (**NOT** a per-file rule modeled on the file-scoped `sync-codex.sh:312-323` rewrites: a per-file rule would leave the token in the files owned by .2/.3 and fail the structural guard at `sync-codex.sh:1627-1633` on those tasks, with the fix unowned). Naming convention: prose role `tracker_runner` (underscore); generated `codex/agents/tracker-runner.toml` (hyphen). Canonical grammar (`operation: <verb> <id>, event: <key>`) is otherwise unchanged.

3. **Two dispatch classes (join-anchored, gap rule 1: "fire-and-forget requires a downstream join; otherwise isolated-but-awaited"):**
   - **Fire-and-forget** — comment-shaped ops whose skill HAS an end-of-run `sync check`: `work.done` (phases.md:298-316) and completion review (phases.md:407-448) both land in the work skill, whose Phase 5 join (phases.md:506-546) is their backstop. The loop continues immediately; a dead dispatch = an unwritten receipt = caught by the pre-join `sync check` → retro-fire once. **Linked-spec precondition:** a comment op is fire-and-forget ONLY when the spec already has a `tracker.id` — on an UNLINKED spec, tracker-sync's create-if-unlinked (steps.md Phase 3) makes the "comment" state-mutating (writes tracker id + merge-base pair + lastSyncedAt), so the host classifies it **state-shaped** for that first touch (awaited + serialized per item 4); two overlapping comment dispatches on an unlinked spec would race to create duplicate issues / torn link state.
   - **Isolated-but-awaited** — comment-shaped ops whose skill has NO end-of-run check: `resolvePr` (workflow.md:437-455) and `qa` (workflow.md:590-606). They dispatch subagent-isolated (context win preserved) but the host AWAITS the outcome line before the skill's summary — seconds cost, keeps the receipt-backstop, and they are end-of-skill anyway.
   - **State-shaped (always awaited)** — reconcile/push touching body merge-base, status, links: `capture` (workflow.md:788-805), `interview` (SKILL.md:462-483), `plan` (steps.md:533-548), `work.firstClaim` (phases.md:205-223), `makePr` (create-and-finalize.md:527-583, the §5.6 `reconcile` op only — §4.6a:240-267 is inline flowctl probing, NOT a fork candidate). A torn merge-base *pair* costs more than the seconds saved (feasibility T3 lost-update hazard), so state-shaped ops are always awaited.

4. **Per-spec serialization + the in-context ledger (R3).** The host keeps a tiny per-conversation ledger (spec id → outstanding agent handle) in its own context. At most ONE in-flight **state-shaped** op per spec id: a new state-shaped dispatch first AWAITS any outstanding dispatch on its spec (this also covers the cross-skill straddle — e.g. a make-pr reconcile that overlaps a still-settling work.done). Comment-shaped dispatches on LINKED specs may overlap freely — even same-spec — because receipts are one-file-per-receipt (feasibility T3: no serialization ever); a comment op on an UNLINKED spec is state-shaped (create-if-unlinked writes link state — item 3) and takes the serialized path until the link exists. The host classifies by the ledger / a cheap `sync get-state` probe. Different specs always parallelize. The host loop is the single serializer — NO flowctl locking machinery. The ledger is per-conversation host context (survives sequential skill invocations in-session); cross-PROCESS serialization is explicitly out of scope (documented limitation, consistent with the repo's multi-agent conventions). Receipts-on-disk are the recovery truth after compaction (edge case below).

5. **Pre-audit join (R4).** Before ANY end-of-run `sync check` — work Phase 5 (phases.md:506-546, check at :528), make-pr §5.7 (create-and-finalize.md:587-609, check at :596), capture Phase 6 (workflow.md:855-882) — the host AWAITS all outstanding dispatches for the audited spec, then the check reads settled receipts. No false MISSING, no duplicate retro-fire. Retro-fire itself stays exactly one cycle, unchanged. The audit race is REAL, not defensive (feasibility "Audit race — demonstrated").

6. **Mode-threading + autonomy parity (R11).** The forked runner invocation carries an explicit `DISPATCH: forked` input; tracker-sync Phase 0 (steps.md:21-30) folds it into the single `RALPH` queue gate: `forked ⇒ queue-not-ask`, `AskUserQuestion` is NEVER reachable in a forked dispatch (parity with the existing R14 invariant). The SKILL.md:45 "Inline skill (no `context: fork`)" claim is NARROWED to: *ceremonies + manual/interactive conflict resolution stay inline; lifecycle dispatches may run forked.*

7. **Interactive conflicts via host-side ask-after-join (R6).** A forked dispatch that hits a genuine conflict QUEUES (`sync defer`) in both modes — it never asks. The host reads the outcome line after the join and — **interactive only** — MAY surface the queued conflict via `AskUserQuestion` at the touchpoint. This preserves today's interactive UX with zero `AskUserQuestion`-in-fork. The manual `/flow-next:tracker-sync reconcile <id>` path stays fully inline and may ask, unchanged.

8. **Config leaf (R5).** `tracker.dispatch` = `async` (default) | `inline`, a top-level scalar sibling of `readyState` in `get_default_tracker_config` (flowctl.py:1131-1140), with a `TRACKER_DISPATCH_MODES` enum constant beside `TRACKER_TIEBREAKS` (flowctl.py:1033). String-enum discipline: the gate runs inline iff `dispatch == "inline"` (exact-match), else async — bool never activates the hatch. `inline` restores today's behavior wholesale (support/debug hatch); the leaf gates dispatch MODE only — gating predicates, ops, receipts, event tags, audit are byte-identical in both modes.

9. **v1 async is Claude-Code-gated (R8).** Codex `spawn_agent` true-async is UNVERIFIED, and the Codex mirror convention forbids background jobs (`sync-codex.sh:421` "Do NOT use background jobs"). So a host-capability check (`host_is_claude_code`, mirroring codex-delegation's gate) resolves async ONLY on Claude Code; **Codex + Copilot + Droid all run inline in v1**. This kills the Codex-async risk. The Codex mirror still gets the `Task flow-next:tracker-runner` → role rewrite (the global sweep in item 2) for prose consistency, but its capability gate resolves to inline. Codex-async verification is future work.

## API Contracts

- No flowctl transport changes; no new receipt fields. Receipts / `sync defer` / `sync check` behave identically — only WHERE the tracker-sync body executes changes.
- New config leaf `tracker.dispatch` (`async` default | `inline`), schema default `async`, `TRACKER_DISPATCH_MODES = {"async","inline"}`, documented in the flowctl.md config table (after the `tracker.readyState` row, L661).
- New agent `agents/tracker-runner.md`: `model: sonnet`, `disallowedTools: Task` (KEEPS Write/Edit — the reconcile writeback needs them — plus Bash; precedent `agents/pr-comment-resolver.md`). Output contract: exactly one line `<status> <spec-id> <note>`, `status` = the full tracker receipt enum `pushed|pulled|merged|updated|diverged|queued|errored|noop`. The agent contract also defines the canonical dispatch+join block (launch background vs awaited, ledger entry, join, timeout handling) that .2/.3 reference instead of re-deriving. `sync-codex.sh` `sandbox_for()` gains `tracker-runner` → `workspace-write` (today only `worker|plan-sync` are write-sandboxed; a read-only generated `tracker-runner.toml` would contradict the Write/Edit-keeping contract — dormant on Codex v1, but must not lie).
- The runner subagent is the EXISTING `flow-next-tracker-sync` skill body driven in fork context with the op/event/spec inputs; no second implementation. Canonical dispatch grammar unchanged: `operation: <verb> <id>, event: <key>` (+ `DISPATCH: forked`).
- Caller-visible contract: fire-and-forget touchpoints (`work.done`, `completionReview`) return immediately after dispatch (outcome line arrives asynchronously, surfaced in the host summary); isolated-but-awaited (`resolvePr`, `qa`) and all state-shaped touchpoints return after the awaited outcome line.

## Edge Cases & Constraints

- **Awaited-runner bound = 10 minutes (R12).** Mirrors the Foreground-rule generous-timeout guidance. On timeout / runner death of an AWAITED op, the host treats it as transport-unreachable → an `errored` receipt (written host-side, event-tagged — the touchpoint ran and failed, matching today's inline semantics), continues, never blocks beyond the bound. **AWAITED ops only:** `sync check` clears on ANY event-tagged receipt (flowctl docstring: "the check asserts the touchpoint ran"), so for a FIRE-AND-FORGET death the host must write NO receipt — the unwritten receipt is exactly what makes the pre-join audit report MISSING and retro-fire once. A host-side event-tagged receipt on a fire-and-forget death would silently kill the backstop.
- Subagent MCP availability: interactively-authed MCP servers may be absent in headless runs — the runner inherits the same transport-ladder degradation (MCP → GraphQL/env-token → no-op + receipt) the inline path has; no new failure mode (feasibility T2 confirmed MCP works in-fork).
- A fire-and-forget dispatch that dies (agent error) = an unwritten receipt = caught by the pre-join `sync check` → retro-fire once (the existing backstop becomes the retry path). Comments are marker-deduped so a duplicate retro-fire is harmless.
- Ordering: firstClaim (state-shaped, awaited) always lands before the first done-comment can dispatch — natural ordering preserved without extra machinery.
- The host ledger must survive context compaction: on resume, treat outstanding handles as unknown → run the pre-audit join via `sync check` semantics (receipts on disk are the truth, not the ledger).
- Hosts without background dispatch (Codex/Copilot/Droid in v1): the capability gate resolves inline — zero behavior change there.
- Never fork the discovery ceremony, link/unlink ceremonies, or `--dry-run`/manual runs — user-facing interaction lives host-side.
- Missing `event:` tags block the end-of-run audit and are fixed in scope (R9): resolve-pr workflow.md:449 (`operation: comment <spec-id>` → add `event: resolvePr`), interview SKILL.md:472 (add `event: interview`), plan steps.md:541 (add `event: plan`), qa workflow.md:602 (the `:` placeholder becomes an EXPLICIT dispatch block with `event: qa` — memory: dispatch instructions must be explicit blocks, never `:` placeholders).
- **completionReview event-key mismatch — FIXED in 2.9.1 (shipped 2026-07-09, ahead of this spec).** The `work.completionReview` tag vs top-level `tracker.perEvent.completionReview` leaf mismatch (dead `sync check` backstop) was released as a standalone patch with regression coverage in `test_sync_check.py` (key-parity class + prose guard). fn-89's only remaining obligation: keep the tag `completionReview` (never `work.`-prefixed) when rewiring the dispatch blocks — the prose-guard test fails the suite if the old shape is reintroduced.

## Foreground rule reconciliation (fn-78)

Host-level background `Task` dispatch — where the harness re-invokes the HOST on the subagent's completion (validated live 2026-07-05) — is DISTINCT from `Bash run_in_background` inside a subagent context (the fn-78 lesson the Foreground rule encodes: a backgrounded bash job inside a worker stalls because nothing re-drives it). Corollary (R13): tracker-runners are dispatched from HOST contexts ONLY — never from worker subagents (workers have no `Task` tool anyway). The spec and `docs/ralph.md` both carry this paragraph so the two mechanisms are not conflated in future work.

## Feasibility validation (dogfooded 2026-07-05, this repo)

Every load-bearing mechanic was exercised live before planning; findings are binding design inputs:

- **T1 — state-shaped op in a background subagent (GraphQL rung):** a sonnet runner executed fn-89's own flow-first push end-to-end (issue FLOW-32 created, `set-tracker-id`, paired `set-merge-base`, `set-last-synced`, receipt) — 6/6 steps, 39s, ~54k tokens absorbed in the runner's context, exactly ONE line returned to the conductor. The same op done inline earlier the same day cost the host the full ~10k-token MCP body echo plus machinery.
- **T2 — MCP rung inside a subagent:** the runner loaded the Linear MCP tool via ToolSearch and posted a lifecycle comment (marker-deduped) + an event-tagged receipt — MCP works in fork context and its response echo lands in the runner, not the conductor (~52k tokens isolated).
- **Audit race — REAL, demonstrated:** `sync check --events work.done` run while T2 was in flight returned `missing:["work.done"]`; after the join it returned `missing:[]`. The pre-audit join (R4) is load-bearing, not defensive.
- **T3 — concurrency semantics:** receipts are one-file-per-receipt (`sync-runs/sync-<spec>-<µs-timestamp>.json`) — two deliberately concurrent receipt writes both landed; NO serialization needed for receipts ever. Spec-JSON writes go through `atomic_write` (temp+rename) — two concurrent `set-last-synced` produced no corruption (file intact, valid JSON), but read-modify-write means concurrent state-shaped ops are a LOST-UPDATE hazard (a torn merge-base *pair* is the harm case) — hence R3's state-shaped-only serialization.
- **Runner tier:** sonnet-class was sufficient for both shapes (mechanical execution + light judgment) — the runner should default to a cheap tier, not inherit the conductor's model (its own routing win).
- **Join mechanics (Claude Code):** background dispatch = `Task` with background execution; the join = blocking result retrieval / the completion notification. Hosts without background dispatch degrade to inline per R8 (Copilot/Droid unchanged).

## Early proof point

Task .1's **first step (Step 0)**, run BEFORE any fn-89 code is edited, is a LIVE interleave + fork-writeback validation. **Step 0 is HOST-executed:** the `/flow-next:work` conductor runs it in HOST context before dispatching the .1 worker — workers carry `disallowedTools: Task` and per R13 runners are never dispatched from worker subagents, so the worker CANNOT own this proof; the conductor records hold/no-hold and the worker consumes the recorded result. **Disposable target:** run the reconcile against a throwaway spec + issue (create a scratch spec, link it, mutate the issue, reconcile, then unlink + delete) — or snapshot/restore the target — so a failed writeback never corrupts a real spec or tracker issue. It needs NO fn-89 code — it uses generic background `Task` dispatch of the EXISTING tracker-sync skill body plus an intervening foreground `Task`, pure live validation:

1. Dispatch a background runner op that is a **reconcile** (tracker-side-changed → spec writeback). This exercises the fork-context Write/Edit writeback of the merged flow body to `.flow/specs/<id>.md` (`references/body-merge.md` Step 5) — which feasibility T1/T2 did NOT prove (they proved only push + comment in fork), and which the `disallowedTools: Task`-only runner contract depends on.
2. While it is in flight, spawn an intervening foreground `Task` (e.g. a scout).
3. Join the runner; confirm the outcome line + receipt + the spec-writeback all settle correctly.

Feasibility T1/T2 validated isolation + await and the audit race, but NOT the interleave (background dispatch → intervening foreground spawn → later join) that the work loop actually exercises, NOR the fork-context reconcile writeback. **If the interleave does not HOLD** (background dispatch survives an intervening foreground `Task` and joins cleanly) **or the fork writeback fails: STOP — land no code and escalate to the user.** The state-shaped serialization model (.2) and the Write/Edit-keeping runner contract both depend on these holding — so they are proven at .1 Step 0, before any code, not discovered at .2.

## Decision Context

"Parallelize everything" was rejected: per-spec sync-state (merge-base pairs, lastSyncedAt) has interleaving hazards, and the measured win isn't concurrency between tracker calls — it's (a) getting seconds of API latency off the work loop's critical path and (b) keeping ~10–17k-token tracker payloads/refs out of the conductor's context (both observed in the fn-88 run, 2026-07-05). Fire-and-forget is confined to append-only marker-deduped comment ops whose skill ALSO has an end-of-run `sync check` + retro-fire backstop that doubles as the retry path; comment ops without that backstop (resolve-pr, qa) stay isolated-but-awaited so the receipt still settles before the skill's summary; state-shaped ops keep await semantics because a torn merge-base snapshot costs more than the seconds saved. The host-as-serializer (in-context ledger + join points) was chosen over flowctl-side locking: the conductor is already the single dispatcher, and receipts-on-disk remain the recovery truth after compaction. v1 gates async to Claude Code (host_is_claude_code) because Codex spawn_agent true-async is unverified and its mirror convention forbids background jobs — a conservative default that eliminates the cross-platform risk while keeping the Codex prose consistent via the sync-codex role rewrite. The `inline` hatch exists because background failures are harder to debug live; flipping one leaf reproduces today's behavior exactly. The design was validated empirically before planning (see Feasibility validation): both transport rungs work in fork context, the token-isolation win is measured (~50k/op absorbed), the audit race is demonstrated rather than hypothesized, and the receipt store's per-file shape lets comment-shaped dispatches skip serialization entirely.

**Re-review 2026-07-09 (fn-90 investigation side-product):** five independent cross-backend reviews (3× cursor, 2× codex — `.flow/artifacts/fn-90-baseline/`) surfaced overlapping Majors against the original plan; the confirmed ones are folded in above — Step 0 host-execution + disposable target, unlinked-comment = state-shaped, full receipt enum on the runner outcome line, awaited-only host receipts (fire-and-forget deaths stay receipt-less), the `sandbox_for` workspace-write addition, and the completionReview event-key normalization (live bug).
